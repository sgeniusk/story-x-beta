# Story X — 만화 제작실 설계 초안

1단계 D의 남은 갈래(만화·웹툰)를 위한 설계 초안이다. 외부 스킬 `aitoon`(조남경, 저작권 보호 창작물)의 파이프라인을 **구조적 레퍼런스로만** 참고했다. aitoon의 파일·텍스트·SVG·HTML은 복제하지 않으며, 단계 흐름과 발상만 가져와 Story X의 자체 개념(작가진·바이블·품질 게이트·전환 패킷)으로 다시 설계한다.

## 배경

지금 Story X는 소설·에세이 갈래는 생성·검토가 닫혀 있지만, 만화 매체는 인터뷰 질문(`projectIntake.ts` comics)과 작가진 페르소나(storyboard-agent, speech-bubble-agent)만 있고 **편집 경험이 비어 있다**. 만화의 원고는 산문이 아니라 컷 시퀀스이므로, 편집 트랙이 산문 textarea여서는 안 된다.

## 레퍼런스 매핑 — aitoon 흐름 → Story X 개념

| aitoon 단계 | 하는 일 | Story X 대응 |
|---|---|---|
| stage1 | 스토리 → 씬 리스트 → 컷 구성 → 레이아웃 | 만화 편집 트랙 = 컷 보드 |
| stage1-2 | 연출 강화·검토 | 작가진 검토(연출가·말풍선 연출가) |
| stage2 | 인계서 → 이미지 프롬프트 → design_brief | 시각 프롬프트 단계 + 시각 바이블 |
| 인계서로 단계 분리 | 컨텍스트 비대화 방지 | Memory Bank · 매체 전환 패킷 |

aitoon은 Midjourney+ChatGPT 외부 도구에 의존하지만 Story X는 Claude 구독을 쓴다. 그리고 완성 이미지 생성은 후속 단계(`getFocusedServiceScope().later`)이므로, 이번 범위는 **이미지 프롬프트와 레이아웃까지**다.

## 1. 만화 편집 트랙 — 컷 보드

소설의 산문 textarea 자리에 **컷 보드**를 둔다. 회차는 컷의 시퀀스다. 컷 하나는 다음을 가진다.

- 기능 — 이 컷이 회차에서 하는 일(상황 제시·강화·전환·반전 등)
- 구도/카메라 — 앵글, 거리, 시선 흐름
- 대사/말풍선 — 컷당 핵심 대사, 말풍선 위치 메모
- 시각 메모 — 표정·소품·배경 변화

스크롤 웹툰은 세로 리듬, 4컷 인스타툰은 네 박자 기능을 기본 템플릿으로 둔다(인터뷰의 `storyboard-rhythm` 답에 따라 분기).

## 2. 바이블 트랙 — 시각 바이블 추가

기존 바이블 섹션(개요·캐릭터·세계관·캐논·문체·승인)에 **시각 바이블**을 더한다.

- 캐릭터 시각 앵커 — 실루엣, 의상, 색, 표정 규칙
- 배경 — 재사용 장소 구조와 동선
- 무드 — 색·조명·렌즈 감각

인터뷰의 `comics-character-anchor`·`visual-continuity` 답이 이 섹션의 초기값을 채운다.

## 3. 제작 3단계

1. **컷 분해** — 회차 스토리/원고 → 씬 → 컷 시퀀스. 각 컷의 기능을 먼저 채운다.
2. **연출 검토** — 연출가(storyboard-agent)가 컷 리듬·스크롤 후킹·읽기 순서를, 말풍선 연출가(speech-bubble-agent)가 말풍선 밀도와 표정 가림을 분리 검토. 작가진 분리 검토 루프를 그대로 쓴다.
3. **시각 프롬프트** — 각 컷 → 이미지 생성 프롬프트. 시각 바이블 앵커를 프롬프트에 주입해 캐릭터 일관성을 지킨다. 결과를 design_brief 형태의 전환 패킷으로 묶는다.

## 4. 작가진 배정

- 연출가(storyboard-agent) — 컷 리듬, 페이지/스크롤 호흡
- 말풍선 연출가(speech-bubble-agent) — 말풍선·침묵 위치
- 쇼러너·캐릭터·연속성 — 컷 보드의 이야기 기능을 기존 시선으로 검토

## 5. 출간 패키지

만화의 플랫폼 증거는 **첫 3컷** — 훅과 저장 포인트를 검증한다(랜딩의 매체 전환 카드와 일치).

## 6. 범위 경계

- 이번 범위 — 컷 보드, 시각 바이블, 연출 검토, 이미지 프롬프트, design_brief 패킷.
- 후속 — 완성 이미지 렌더링, 컷별 영상 전환.
