# P2-d 응결 목표 분량 설계

> 기준 브랜치: `codex/p2-choice-composer` · 구현 브랜치: `codex/p2-condense-target-length`
> 사용자 결정: 다음 개발을 순차 진행. 주 결제 페르소나의 회당 5천자 요구를 기본값으로 삼되, 짧고 긴 회차도 회차별로 선택할 수 있게 한다.

## 1. 문제와 제품 결정

현재 PLAY 응결은 프롬프트 안의 `1,800~2,700자` 고정 문구만 가진다. 사용자는 생성 전에 목표를 알거나 바꿀 수 없고, 잡·보관함·복구본·WRITE도 그 의도를 기억하지 못한다. 이는 “회당 약 5천자를 마감하는 한국어 장편 연재 작가의 개인 스튜디오”라는 Story X의 핵심 약속과 맞지 않는다.

P2-d는 다음을 하나의 계약으로 도입한다.

- 새 응결의 기본은 **5천자**다. 3천자·5천자·8천자 중 회차마다 선택한다.
- 선택값은 강제 할당량이 아니라 회차 목표다. 연속성·원문 사실·후크가 분량보다 우선한다.
- 목표 미달·초과는 생성 실패가 아니다. 결과를 보존하고 정직한 검토 경고를 낸다.
- 자동 패딩·자동 절단·자동 재생성·목표 자동 하향은 하지 않는다.
- 생성 시작 순간의 목표와 source span을 함께 고정한다. 재시도도 같은 원문과 같은 목표를 쓴다.
- 승인된 회차는 자기 목표를 기억해 WRITE 진행률과 사후 분석이 고정 5천자 가정을 쓰지 않게 한다.

## 2. 도메인 계약

시즌 총 화수용 `StoryContract.lengthClass`와 이름이 충돌하지 않도록 별도 `EpisodeLengthContract`를 둔다.

```ts
type EpisodeLengthPreset = 'compact' | 'standard' | 'extended';

interface EpisodeLengthContract {
  schema: 'storyx/episode-length/v1';
  preset: EpisodeLengthPreset;
  targetChars: number;
  minChars: number;
  maxChars: number;
  generationMinChars: number;
  generationMaxChars: number;
  minScenes: number;
  maxScenes: number;
}
```

| UI | preset | 목표 | 결과 허용 범위 | 생성 안전 목표 | 압력이 달라지는 장면 |
|---|---|---:|---:|---:|---:|
| 3천자 | `compact` | 3,000 | 2,700~3,300 | 2,850~3,150 | 2~3 |
| 5천자 · 기본 | `standard` | 5,000 | 4,500~5,500 | 4,750~5,250 | 3~4 |
| 8천자 | `extended` | 8,000 | 7,200~8,800 | 7,600~8,400 | 4~6 |

글자 수는 WRITE의 기존 사용자 표면과 일치시켜 **본문 `prose`에서 모든 공백 문자를 제외하고 문장부호는 포함**해 센다. 제목·후크·outline·beats·캐논 후보는 세지 않는다.

평가는 `under | within | over` 세 상태다. 실제 글자 수는 결과 본문에서 결정론적으로 다시 계산할 수 있지만, 새 CLI 결과에도 요청 계약·실제 글자 수·상태를 함께 넣어 생성 당시 영수증을 자기완결적으로 만든다.

## 3. 생성·품질 계약

프롬프트의 분량·장면 문장은 계약값으로 동적 생성하되 TypeScript 기준 빌더와 실제 `tools/storyx.mjs` 미러는 전체 결과를 byte-identical로 유지한다.

분량과 무관하게 다음 불변식을 유지한다. 여기서 **원문 근거**는 `응결할 대화`에 명시된 사건·행동·결정·대사다. 현재 장면·캐릭터·기존 기억·이야기 아크는 원문을 해석하고 모순을 막는 제약이며, 새 사건을 일으키는 재료가 아니다.

- 장면 수와 시도 → 반격 → 비용 → reveal → 강제 선택 연쇄는 원문이 각 단계를 뒷받침할 때만 적용한다. 부족하면 더 적은 장면과 짧은 본문으로 멈춘다.
- 목표 분량·장면 수·후크를 위해 새 행동·결정·사건·폭로·대가·페이오프를 만들지 않는다. 새 서브플롯·세계 규칙·과거사·이름·기관·범행 원인도 발명하지 않는다.
- 입력에 있는 행동·감각·갈등 대사를 먼저 구체화한다. 그래도 부족하면 짧은 결과를 반환한다.
- 질문의 답·다음 질문·말미의 위협·시한·강제 선택도 원문에 있을 때만 사용한다. 없으면 이미 일어난 가장 강한 행동이나 감각에서 끝낸다.
- 후크 뒤에 분량 보충 문장을 붙이지 않는다.
- 기존 캐논 충돌·누수 차단·사용자 승인 전 미반영 계약을 약화하지 않는다.

새 응결은 캐논화 차단 verdict가 붙은 턴을 제외한 유효 메시지가 기존 최소치보다 많을 때만 시작한다. 차단 턴만 쌓인 상태에서는 잡을 만들지 않는다.

## 4. 영속·복구·승인 경계

`EpisodeLengthContract`는 다음 네 곳에 같은 값으로 보존한다.

1. `DiveCondenseJobRequest` — 잡 dedupe와 실제 CLI 입력의 정본.
2. `PlayRecoverySnapshot` — 등록 실패·취소·만료 뒤 동일 시도 복구.
3. `GenerationInboxItem` root — 성공 영수증에서 recovery가 용량 압축돼도 목표 보존.
4. 승인된 `Chapter` — WRITE와 StoryScore가 그 회차의 목표를 사용.

새 요청에서는 계약이 필수다. 저장된 구버전 영수증·복구본·회차에서는 optional로 읽는다. 누락된 과거 성공 결과에 새 목표를 소급 표시하지 않는다. 손상된 optional 계약은 그 필드만 버리고 본문·원문·영수증·체크포인트는 보존하되, 자동 재시도에서 기본 5천자로 대체하지 않는다.

재시도는 실패 영수증 root의 `sourceSpan`, `scene`, `episode`, `EpisodeLengthContract`를 사용한다. 영수증이 있으면 root 목표만 정본으로 인정하고 recovery 목표로 보충하지 않는다. 현재 PLAY에서 frozen span의 전체 ordered ID와 연속 turn이 정확히 일치하며 차단 턴을 거른 원문이 비어 있지 않을 때만 자동 재시도한다. legacy·누락·손상·부분 span 또는 목표는 TXT와 분리 복구 작업본으로만 구제한다. 잡 등록 전에 로컬에서 실패해 영수증 자체가 없을 때만 같은 recovery snapshot의 목표를 쓴다. 재시도 사이 추가된 PLAY 턴은 이번 transcript 밖에 남아 다음 회차 재료가 된다. 결과 보류·실패·취소·길이 경고는 source를 소비하지 않으며, 승인 성공 뒤에만 기존 exact checkpoint 계약대로 한 번 소비한다.

root 목표가 삭제됐더라도 result의 `episodeLength`·`actualChars`·`lengthStatus` 또는 recovery의 목표가 남아 있으면 현대 영수증 marker만 복원한다. nested 목표를 root 정본으로 승격하지 않고 승인·재시도·복구 저장을 막는다. 분리 작업본은 commit 직전뿐 아니라 write-ahead journal 저장본을 다시 읽은 뒤에도 durable receipt의 id·project·episode를 재검사하고 root 목표로 다시 정본화한다.

로컬 Codex의 5천~8천자 장문 응결은 provider 9분, 잡 10분의 분리된 제한을 쓴다. provider 제한 초과는 같은 장문 호출을 맹목 재시도하지 않고 원문·목표를 보존한 채 `timed-out` 진단만 남긴다. stderr 원문은 사용자 화면이나 영수증에 노출하지 않는다.

직접 API와 CLI도 `transcript`가 비어 있거나 문자열이 아니거나 CLI에서 다음 flag를 값처럼 읽은 경우 생성 전에 거부한다.

승인 시 `ProductionRequest.episodeLength`를 통해 Chapter에 계약을 기록한다. 실제 글자 수는 편집 후 달라질 수 있으므로 Chapter에 별도 고정 숫자로 중복 저장하지 않고 현재 `prose`에서 다시 계산한다.

## 5. 사용자 경험

PLAY 하단 sticky dock 바로 앞에 비고정 `회차 분량` 계약 레일을 둔다. 응결 가능한 새 재료가 있을 때만 나타나며, 기존 `지금 응결` 버튼도 작성창에서 이 레일로 옮겨 대화 작성과 회차 생성의 목적을 분리한다.

- native fieldset/radio로 `3천자 · 2~3장면` / `5천자 · 3~4장면` / `8천자 · 4~6장면`
- 보조 문구: `재료가 모자라면 짧게 돌아오며, 자동으로 내용을 만들거나 잘라내지 않습니다.`
- CTA: `5천자로 응결`처럼 현재 선택을 다시 말한다.
- 선택은 해당 프로젝트의 PLAY 세션에 남는다. 새 세션과 구버전 세션은 5천자로 시작한다.
- 잡 등록·실행·결과 검토 중에는 선택을 잠가 실행 중 계약과 화면이 어긋나지 않게 한다.

생성 중 카드는 `5천자 목표 · 3~4장면 응결 중`처럼 표시한다. 보관함 카드도 목표를 표시한다.

검토창은 본문 위에 다음 중 하나를 표시한다.

- `4,820자 / 목표 5,000자 · 범위 안`
- `3,940자 / 목표 5,000자 · 목표 미달, 자동으로 채우지 않음`
- `5,810자 / 목표 5,000자 · 목표 초과, 자동으로 자르지 않음`

미달·초과도 승인할 수 있다. 이때 승인 버튼에 실제 글자 수를 다시 표시해 사용자가 경고를 보고 선택했음을 분명히 한다. WRITE의 이번 회차 진행률은 Chapter의 목표를 분모로 쓴다. legacy 회차는 기존 5천자 표시를 유지하되 새 계약이 있었던 것처럼 보관함에 소급 표기하지 않는다.

## 6. 비목표와 경계

- 목표를 맞추기 위한 다단계 확장·축약 생성은 하지 않는다.
- 모델이 반환한 미달·초과를 자동 실패 처리하지 않는다.
- 이번 슬라이스에서 캐논 후보별 근거 ID·opt-in, retcon 재구성 회차, 동시 후보 supersede 모델을 새로 설계하지 않는다. 이는 별도 캐논 승인 신뢰성 슬라이스다. P2-d는 기존 사용자 검토·승인과 source checkpoint를 그대로 보존하고, 장문을 위해 근거 없는 캐논을 만들라는 지시는 절대 추가하지 않는다.
- 전체 시즌 화수 계약이나 온보딩 분량 질문을 바꾸지 않는다.

## 7. 수용 기준

1. 새 PLAY 세션은 5천자가 선택되어 있고 3천·8천으로 바꾼 값이 세션 저장 후에도 유지된다.
2. 각 프리셋이 요청 → dedupe → CLI → 영수증 → 복구 → 검토 → Chapter에 같은 v1 계약으로 도달한다.
3. 다른 목표는 다른 잡으로 취급되고, 실패 재시도는 현재 선택이 아니라 실패 시도의 frozen 목표·source를 사용한다.
4. 차단 verdict 메시지만으로는 잡이 등록되지 않는다.
5. TS/CLI 프롬프트가 세 프리셋 모두 byte-identical이며 각 범위·장면 수·무패딩 문구를 포함한다.
6. under/within/over가 공백 제외 본문 글자 수로 결정되고, 미달·초과도 보존·검토·명시 승인 가능하다.
7. 승인 회차의 WRITE 진행률과 StoryScore가 해당 회차 목표를 사용한다. legacy 입력은 깨지지 않는다.
8. 390px와 320px 폭에서 선택기·CTA·작성창·동작 버튼·검토 경고가 잘리거나 겹치지 않고, 선택기와 CTA의 터치 높이가 44px 이상이다.
9. 기존 exact source 소비, stale 경고, 누수 차단, 캐논 검토, 보관함·복구 경로의 회귀가 없다.
10. 누락·부분·차단-only source와 손상·누락 root 목표는 자동 재시도하지 않으며, 직접 API/CLI의 비문자·공백·누락 원문도 생성 전에 실패한다.
