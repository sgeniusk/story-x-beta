# P0-b 응결 실패 구제 — PLAY 기록을 잃지 않는 복구 경로 (2026-07-16)

## 문제

P0-a는 응결을 잡+폴링으로 분리해 화면 이탈에는 강해졌지만, 잡 시작 실패·생성 실패·취소·시간 초과·서버 재시작 만료 뒤에는 `폐기` 외의 다음 행동이 없다. 실제 플레이테스트에서는 긴 PLAY를 마친 사용자가 결과 생성에 실패한 뒤 대화 원문까지 잃었다고 느꼈다. P0-b의 목적은 생성 성공률을 다시 다루는 것이 아니라 **생성에 실패해도 사용자가 만든 PLAY 기록을 즉시 가져가거나 원문을 참고해 직접 쓸 수 있게 하는 것**이다.

## Brainstorm — 설계 결정 질문과 답

### 1. 무엇을 복구 단위로 볼 것인가?

- 응결용으로 잘라낸 span이 아니라 사용자가 잡을 시작한 순간의 PLAY 전체 기록이 복구 단위다.
- `<AI>장면·작품명·예정 회차·전체 대화 원문을 하나의 `PlayRecoverySnapshot`으로 잡 시작 전에 캡처한다. 이후 PLAY가 바뀌어도 실패 영수증은 당시 원문을 보존한다.</AI>`

### 2. 어느 실패에서 구제 행동을 보여 줄 것인가?

- 잡 등록 자체가 실패하면 현재 PLAY 안에서 바로 보여 준다.
- 생성 보관함에서는 `failed`·`timed-out`·`expired`·`cancelled` 영수증에 보여 준다.
- `<AI>취소도 생성 결과는 없지만 사용자가 만든 원문은 다시 쓸 가치가 있으므로 같은 복구 경로를 제공한다. 성공 영수증은 기존 정밀 검토·승인 경로를 유지한다.</AI>`

### 3. 어떤 결과물로 구제할 것인가?

- `PLAY 기록 TXT` — 로컬 텍스트 파일로 즉시 내려받는다.
- `원문으로 직접 쓰기` — 원래 `projectId`의 **복구 작업본**을 열되 아직 본편 회차를 만들지 않는다.
- `<AI>TXT에는 작품·회차·장면·보존 시각·역할 라벨이 붙은 원문을 담는다. WRITE에서는 PLAY 원문을 별도 참고 패널에만 두고, 비어 있는 작업 본문에서 사용자가 직접 소설 문장으로 다듬는다.</AI>`
- `<AI>작업 본문 아래의 명시적 `회차로 저장`을 눌렀을 때만 `Chapter`를 하나 만들고 본편·회차 수·품질 지표·출간 입력에 반영한다.</AI>`

### 4. 캐논과 메모리는 어떻게 보호할 것인가?

- 복구는 생성 결과 승인도, 캐논 승인도 아니다.
- 복구 작업본은 `SeriesProject.chapters` 밖에 있으므로 열기·타이핑·새로고침만으로 캐논·회차·인물·지표가 바뀌지 않는다.
- 명시적으로 저장한 WRITE 회차의 `newCanonFacts`는 항상 빈 배열이며 잠그지 않는다.
- `<AI>`회차로 저장` 시점에만 `chapterFromDraftPayload`에 사용자가 쓴 제목·본문과 빈 intent·pressure를 전달해 성장 상태를 자동 갱신하지 않고, 기존 캐논 배열과 인물 상태가 그대로인지 순수 테스트로 고정한다.</AI>`
- 앵커 위반 턴도 원문 TXT/초안에는 남는다. 다만 캐논 후보로 승격하지 않는다. 원문 보존과 캐논 승인을 분리한다.

### 5. 중복 전송과 이전 데이터는 어떻게 다룰 것인가?

- `<AI>작업본을 연 시점은 영수증의 `recoveryDraftOpenedAt`·`recoveryDraftId`에 기록하고 버튼은 `직접 쓰던 작업본 열기`로 계속 활성화한다. `recoveredAt`·`recoveredChapterId`는 사용자가 명시적으로 회차 저장을 마친 뒤에만 기록하며, 그때 `회차로 저장됨`으로 비활성화한다.</AI>`
- `<AI>같은 생성 영수증을 다시 열면 프로젝트별 복구 작업본 저장소의 기존 제목·본문을 덮지 않고 그대로 재개한다. 저장 핸들러는 같은 draft id의 연속 클릭을 한 번만 커밋한다.</AI>`
- `<AI>구버전 영수증은 recovery 필드 없이 그대로 파싱한다. 복구 스냅샷이 없는 오래된 영수증에는 불가능한 버튼을 만들지 않는다.</AI>`
- `<AI>이전 구현이 만든 오염 회차는 영수증 chapter id, 제목·hook·원문 포맷, 빈 구조/캐논, 미잠금, 최신 회차, currentEpisode가 모두 일치하고 PLAY working copy도 안전하게 함께 제거할 수 있을 때만 빈 복구 작업본으로 환원한다. 편집·잠금·후속 회차 중 하나라도 있으면 사용자 원고로 보고 자동 변경하지 않는다.</AI>`
- 원래 작품이 보관함에 없으면 다른 작품에 넣지 않고 TXT 보관을 안내한다.

### 6. 본편과 PLAY 작업본이 갈라져 있으면 어떻게 할 것인가?

- `<AI>복구 작업본 열기는 본편과 PLAY working copy를 전혀 바꾸지 않으므로 미반영 항목이 있어도 허용한다.</AI>`
- `<AI>`회차로 저장` 시점에 PLAY에 본편 미반영 회차·캐논이 있으면 자동 병합하지 않고, 작성한 복구 작업본을 보존한 채 ⟳최신화를 먼저 요구한다. 캐논 승인 게이트를 복구 편의로 우회하지 않기 위해서다.</AI>`
- `<AI>미반영 항목이 없으면 그때만 새 회차를 본편에 저장하고 기존 PLAY session은 보존한 채 working project를 같은 본편 스냅샷으로 맞춘다.</AI>`

### 7. 로컬 보관공간이 부족하면 어떻게 할 것인가?

- `<AI>영수증 저장 실패는 이미 시작된 서버 잡의 실패로 간주하지 않는다. 성공 결과·실행 중 잡·미복구 PLAY 원문은 삭제하지 않고, 성공 결과에 중복으로 붙은 복구 snapshot만 줄여 한 번 재시도한다.</AI>`
- `<AI>재시도도 실패하면 메모리 UI에는 기록을 유지하고, 실행 중 상태라도 새로고침 전 TXT 보관 행동과 경고를 함께 표시한다.</AI>`

### 8. 회차 저장 도중 일부 localStorage 쓰기만 성공하면 어떻게 할 것인가?

- `<AI>회차를 만들기 전에 작업본에 `commitIntent(chapterId·chapterTitle·requestedAt)`를 먼저 영속하는 write-ahead 계약을 둔다. 본편 저장 뒤 영수증·작업본 정리가 실패해도 재시도는 intent가 가리키는 기존 회차를 식별하고 새 회차를 만들지 않는다.</AI>`
- `<AI>완료 영수증이 실제 localStorage에 남았고 작업본 제거까지 성공한 뒤에만 일반 WRITE로 돌아간다. 중간 실패 시 작업본은 `저장 마무리` 상태로 남고 제목·본문 편집과 화면 이탈을 잠가 중복·유실을 함께 막는다.</AI>`
- `<AI>legacy 오염 회차 환원도 작업본에 `legacyRepair` journal을 먼저 남긴다. 회차 제거 뒤 receipt 저장이 실패하거나 앱이 닫혀도 다음 시작에서 `기존 journal + 회차 부재`를 확인해 receipt 전환을 재개한다.</AI>`
- `<AI>잡 등록 전에 실패한 복구도 작업본을 처음 열 때 `local-{draftId}` 영수증을 생성 보관함에 만든다. 따라서 작업본을 닫아도 전역 보관함에서 다시 열 수 있다.</AI>`
- `<AI>작업본 진입은 draft 저장과 영수증의 `recoveryDraftId` 영속이 모두 확인된 뒤에만 완료로 본다. local receipt 저장이 실패하면 화면을 열어 둔 채 오류를 표시해 작업본이 전역 보관함에서 고립되지 않게 한다.</AI>`

### 9. 여러 탭과 보관 개수 제한에서도 어떤 작업본을 우선할 것인가?

- `<AI>재열기에서는 클릭한 영수증 id로 새 draft id를 만들기 전에 영수증의 `recoveryDraftId`가 가리키는 기존 작업본을 우선한다. 잡 등록 전 local receipt도 같은 작성 본문으로 돌아간다.</AI>`
- `<AI>저장 시작 시 탭의 메모리 draft가 아니라 저장소의 최신 draft를 다시 읽는다. 최신 `commitIntent`·`legacyRepair`를 오래된 탭 상태로 지우지 않으며, 이미 시작한 거래를 먼저 마무리한다.</AI>`
- `<AI>legacy 오염 회차를 제거하기 직전에도 본편과 PLAY working을 다시 읽어 엄격 판정을 재실행한다. 판정 뒤 다른 탭이 편집한 원고를 오래된 snapshot으로 덮어쓰지 않는다.</AI>`
- `<AI>20개 정리 기준은 완전한 빈 비활성 작업본에만 적용한다. active 작업본, 작성한 제목·본문, `commitIntent`·`legacyRepair`가 있는 작업본은 한도를 넘더라도 보존해 사용자 원고와 복구 journal을 조용히 버리지 않는다.</AI>`

## 사용자 계약

1. 응결을 시작하는 순간 당시 PLAY 전체 기록을 로컬 생성 영수증에 함께 보존한다.
2. 생성 실패·취소·시간 초과·연결 만료 뒤에도 원문 TXT와 직접 쓰는 복구 작업본 두 구제 행동을 제공한다.
3. 잡 등록 자체가 실패해도 현재 PLAY 안에서 같은 두 행동을 바로 제공한다.
4. TXT는 생성된 문장이 아닌 사용자의 PLAY 원문이며, 파일명은 작품명과 예정 회차를 식별할 수 있다.
5. WRITE 진입은 원래 작품만 활성화한 뒤 본편과 분리된 복구 작업본을 연다. 다른 작품으로 조용히 우회하지 않는다.
6. 작업본의 PLAY 원문은 별도 참고 패널에 있고, 본문은 비어 있는 상태로 시작한다. 원문 메타데이터를 소설 본문에 넣지 않는다.
7. 작업본 열기·작성·새로고침은 회차·캐논·인물·성장·품질 지표를 바꾸지 않는다.
8. 사용자가 `회차로 저장`을 누른 뒤에만 편집 가능한 새 회차를 하나 만들며 캐논·인물 상태를 자동 변경하지 않는다.
9. 작업본은 다시 열 수 있고, 회차 저장을 완료한 뒤에만 생성 영수증 버튼을 비활성화한다. TXT 내려받기는 계속 허용한다.
10. 본편 미반영 PLAY가 있으면 회차 저장 시점에 최신화를 요구하며, 작성한 작업본은 보존한다. 안전할 때는 저장 뒤 본편과 PLAY working project의 다음 회차를 일치시킨다.
11. 회차 저장이 부분 성공하면 같은 화면의 `저장 마무리`로 재개하며 같은 본문을 새 회차로 다시 만들지 않는다.
12. 작업본 저장 실패나 저장 마무리 중에는 화면 이탈을 막고, 잡 시작 전 실패에서 만든 작업본도 생성 보관함에서 다시 연다.
13. 여러 탭에서 같은 작업본을 열어도 먼저 영속된 저장 journal을 오래된 탭이 지우지 않으며 같은 본문을 중복 회차로 만들지 않는다.
14. 작업본 정리 한도는 빈 비활성 항목만 제거하며 작성 본문·active·부분 저장 journal은 보존한다.

## 데이터 모델

```ts
interface PlayRecoverySnapshot {
  schema: 'storyx/play-recovery/v1';
  projectId: string;
  projectTitle: string;
  episode: number;
  scene: string;
  transcript: string;
  capturedAt: string;
}

interface PlayRecoveryWorkDraft {
  schema: 'storyx/play-recovery-work-draft/v1';
  id: string;
  projectId: string;
  generationId?: string;
  episodeHint: number;
  title: string; // 빈 값 허용, 사용자가 직접 작성
  body: string;  // 빈 값에서 시작
  source: PlayRecoverySnapshot;
  commitIntent?: { chapterId: string; chapterTitle: string; requestedAt: string };
  legacyRepair?: { chapterId: string; startedAt: string };
  createdAt: string;
  updatedAt: string;
}

interface GenerationInboxItem {
  // 기존 필드 유지
  recovery?: PlayRecoverySnapshot;
  recoveryDraftOpenedAt?: string;
  recoveryDraftId?: string;
  recoveredAt?: string;
  recoveredChapterId?: string;
  approvedCondenseCheckpoint?: {
    chapter: Chapter; // 충돌 선택까지 반영한 실제 저장 회차
    retcons: Array<{ factId: string; previousStatement: string; statement: string }>;
    condensedThroughTurn: number; // 최초 승인 때 실제로 응결한 PLAY turn 경계
    baseProjectRevision: string; // 승인 결정을 만든 직전 본편의 전체 상태 지문
    committedProjectRevision: string; // 승인 결과가 반영된 직후 본편의 전체 상태 지문
  };
  localPersistenceFailed?: boolean; // 메모리 전용, localStorage에는 기록하지 않음
}
```

스냅샷은 기존 생성 보관함 localStorage에 보존한다. 복구 작업본은 프로젝트별 여러 draft와 active draft를 담는 별도 localStorage에 보존한다. 서버 잡 요청과 API 응답 스키마에는 원문 복구 필드를 추가하지 않는다. 구버전 데이터와 서버 계약을 건드리지 않는 클라이언트 구제 계층이다.

## UI

### ProjectHub 생성 보관함

- 실패 계열 영수증 아래에 PLAY 기록은 안전하며 직접 쓰는 복구 작업본도 본편이 아님을 표시한다.
- 행동 순서는 `PLAY 기록 TXT` → `원문으로 직접 쓰기` → `폐기`다.
- 작업본을 연 뒤에는 두 번째 버튼이 `직접 쓰던 작업본 열기`로 바뀌고 계속 사용할 수 있다. 명시 저장 뒤에만 `회차로 저장됨`으로 바뀌고 disabled가 된다.
- 기존 `--nx-*` 라이트 브리지 토큰을 유지한다. 실패색은 상태 라벨에만 쓰고 복구 행동은 차분한 기본/primary 위계를 쓴다.

### PLAY 잡 등록 실패

- 기존 한 줄 오류 아래에 작은 복구 카드를 연다.
- 영속이 확인된 실패는 `응결은 멈췄지만 PLAY 기록은 안전합니다.`로, 미영속 실패는 `PLAY 기록이 아직 보관함에 저장되지 않았습니다.` 경고로 표시하고 `응결 다시 시도`·TXT·수동 작성 행동을 제공한다.
- 기존 `--st-*` 스튜디오 토큰만 사용한다. 새 애니메이션은 만들지 않는다.

### WRITE 복구 작업실

- 원고지 바깥에 `복구 작업본 · 아직 본편 아님` 배너를 둔다.
- 본문은 빈 값에서 시작하며 `작품으로 남길 첫 문장을 직접 써보세요` placeholder를 제공한다.
- 제목과 첫 설명에서 `응결된 원고가 아님`을 밝힌다.
- PLAY transcript는 원고지와 형제 관계인 접이식 `PLAY 원문 보기` 참고 패널에만 표시한다.
- 제목은 선택 입력이며, 비어 있으면 실제 저장 시 현재 다음 회차의 기본 제목(`N화`)을 사용한다.
- `회차로 저장`은 본문이 비어 있으면 비활성화하고, 저장할 때에만 본편 회차·지표에 반영된다는 설명을 함께 둔다.
- `본편으로 돌아가기`는 작업본을 삭제하지 않고 active 표시만 해제한다. 생성 보관함에서 같은 작업본을 다시 열 수 있다.
- 저장 실패 또는 `commitIntent`가 남은 동안에는 `본편으로 돌아가기`를 비활성화한다. 저장 마무리가 끝나야 작업본을 제거하고 일반 WRITE로 이동한다.
- 복원된 `commitIntent`·`legacyRepair`가 있으면 작업실 내부뿐 아니라 PLAY·PLAN·프로젝트·출간 이동과 브라우저 종료에도 같은 이탈 가드를 적용한다.
- 제목 입력의 Enter와 한국어 IME 확정 Enter는 제출로 해석하지 않는다. 명시적 버튼만 회차 저장을 시작한다.
- 스튜디오 warm `--st-*` 토큰과 WRITE 모드색만 사용하고 새 모션은 추가하지 않는다.

## 사용자 테스트 2차 결정 — `이어쓰기` 의미 분리

- 사용자 관찰: 응결 결과가 없는 상태에서 빈 WRITE 복구 작업실을 `이어쓰기`라고 부르면 무엇을 이어 쓰는지 알 수 없고, 응결 기능이 고장 난 것처럼 느껴진다.
- 확인된 원인: 작품 보관함의 `이어쓰기`는 회차 수와 무관하게 항상 editor로 이동하며, 실패 구제 CTA의 `WRITE에서 이어쓰기`는 응결 결과가 아니라 빈 수동 복구 작업본을 연다. 두 행동이 같은 말로 노출돼 의미가 섞였다.
- 질문 1 — 작품 카드의 기본 재개 지점은 어디인가?
  - <AI>카드는 모드를 섣불리 약속하지 않는 `작업 계속하기`로 표시한다. 작성 내용·저장 journal이 있는 복구 작업본은 WRITE를 우선하고, 그렇지 않은 0화 작품에 저장된 PLAY 세션이 있으면 PLAY로 돌아간다. 빈 복구 작업본은 생성 보관함에서 다시 열 수 있을 때만 비활성화해 기본 재개를 가로채지 않게 한다.</AI>
- 질문 2 — 실패 복구 작업본은 무엇이라고 부르는가?
  - <AI>`원문으로 직접 쓰기` / `직접 쓰던 작업본 열기`로 표시한다. 작업실에도 `응결된 원고가 아님`을 첫 설명으로 노출한다.</AI>
- 질문 3 — 응결 실패 뒤 재시도는 자동인가?
  - <AI>자동 생성은 하지 않는다. PLAY 현장에 `응결 다시 시도`를 명시적으로 제공하고 사용자가 눌렀을 때만 새 잡을 시작한다.</AI>
- 질문 4 — 재시도와 과거 실패 중 무엇이 PLAY 현장을 대표하는가?
  - <AI>현재 회차의 `createdAt` 기준 최신 생성 시도 하나만 대표한다. 작업본 연결로 영수증 배열 순서가 바뀌어도 최신 시도가 실행 중이거나 성공했으면 같은 회차의 과거 실패 카드를 다시 띄우지 않으며, 지난 회차 실패는 생성 보관함에만 남긴다.</AI>
- 질문 5 — 원문 영수증이 실제로 저장되지 않았을 때도 안전하다고 말할 수 있는가?
  - <AI>말할 수 없다. `localPersistenceFailed`이면 상태가 실패든 성공이든 안전 문구를 숨기고 새로고침 전 결과 검토·TXT 다운로드를 `alert`로 표시한다. 직접 쓰기 행동은 보존하되 저장 실패 시 기존 거래 가드가 진입을 중단한다.</AI>

이 수정은 성공 응결→검토→승인→WRITE 흐름과 실패 원문→수동 복구 흐름을 언어와 라우팅에서 분리한다. 빈 복구 작업본은 삭제하지 않고 보관함 재열기 연결을 확인한 뒤 active 표시만 해제한다. 회차가 있거나 PLAY가 없거나 작성/저장 중인 복구본이 있으면 WRITE로 안전하게 폴백한다.

## 사용자 테스트 3차 결정 — 승인 결과의 WRITE 합류

- 사용자 관찰: `승인 — 캐논으로 고정`을 눌렀는데 PLAY에는 1화가 보이고 WRITE에는 빈 원고가 보였다. 이전 실패에서 연 빈 복구 작업본도 WRITE를 먼저 가렸다.
- 확인된 원인: 승인은 `DiveState.project`에만 회차를 넣고, 본편 `loadProject()`에는 `⟳ 최신화`를 한 번 더 눌러야 했다. 버튼 언어는 최종 승인인데 저장 계약은 숨은 2단계였다.
- 질문 1 — 승인과 본편 반영을 계속 두 행동으로 둘 것인가?
  - <AI>아니다. 성공 결과 검토 화면의 명시적 승인은 이미 사용자 캐논 게이트다. 승인 후보를 App이 받아 최신 본편과 기존 `deriveReconcilePlan`으로 다시 비교하고, 충돌이 없으면 PLAY working과 본편을 같은 합류 결과로 영속한 뒤 WRITE로 이동한다.</AI>
- 질문 2 — 승인 후보가 최신 본편과 충돌하면 자동으로 덮어쓸 것인가?
  - <AI>아니다. 후보는 아직 영속하지 않고 기존 `ReconcileReview`를 연다. 사용자가 retcon/keep을 확정할 때만 PLAY working·본편을 함께 저장한다. 취소하면 응결 검토 화면과 생성 영수증을 그대로 유지한다.</AI>
- 질문 3 — 생성 영수증은 언제 제거하는가?
  - <AI>충돌 선택까지 반영한 정확한 회차·retcon checkpoint를 영수증에 먼저 영속하고, PLAY working과 본편 영속이 성공한 뒤 마지막에 제거한다. 중간 실패에서는 원본 후보로 재계산하지 않고 checkpoint를 멱등 재개한다. 저장 실패나 충돌 검토 취소 전에는 승인 원문을 보관함에서 잃지 않는다.</AI>
- 질문 4 — 이전 실패의 빈 복구 작업본은 어떻게 하는가?
  - <AI>본문·제목·journal이 비어 있고 전역 보관함으로 재열 수 있을 때만 active 표시를 해제한다. 작업본 자체와 원문 영수증은 삭제하지 않는다. 작성 내용이나 부분 저장 상태는 계속 WRITE를 우선한다.</AI>
- 질문 5 — 승인·폴링 사이 다른 탭의 변경은 어떻게 보호하는가?
  - <AI>승인 요청에는 project뿐 아니라 승인 직전의 전체 PLAY session을 함께 담는다. checkpoint 영속 전과 실제 PLAY 저장 직전에 durable `DiveState`의 session·project가 모두 정확히 같은지 확인하고, 최신 session에 최초 응결 turn 경계만 적용한다. 느린 running 응답과 404는 mutation 시점 영수증이 여전히 running일 때만 반영해 성공 결과·checkpoint를 terminal 상태에서 역행시키지 않는다.</AI>
- 질문 6 — localStorage의 비원자적 read-modify-write를 여러 탭에서 어떻게 막는가?
  - <AI>Chromium 로컬 모드에서는 App보다 바깥에서 origin 단위 exclusive Web Lock을 세션 동안 유지한다. lock을 가진 한 탭만 편집기와 poller를 mount하고, 다른 탭은 작품을 쓰지 않은 채 자동 대기한다. writer 탭이 닫히면 다음 탭이 lock을 이어받는다. Web Locks가 없는 환경은 작품 유실을 막기 위해 편집기를 mount하지 않고 지원 브라우저 안내를 표시한다.</AI>
- 질문 7 — 실패 뒤 직접 쓰기를 연 상태에서 늦은 성공 결과가 도착하면 무엇을 보존하는가?
  - <AI>성공 결과와 직접 쓰던 복구 작업본을 모두 사용자 자산으로 보존한다. 성공 결과 검토 CTA와 함께 작업본 재열기·TXT·미반영 경고를 계속 표시하고, 해당 영수증 폐기 전 확인을 받는다.</AI>
- 질문 8 — checkpoint 뒤 부분 실패에서 PLAY를 어떻게 재개하는가?
  - <AI>일반 최신화와 분리된 전용 `diveStateVersion`으로만 PLAY를 durable state에서 remount한다. 이로써 같은 화면의 멱등 재승인은 가능하게 하되 일반 최신화가 전송 전 입력이나 검토 결정을 지우지 않게 한다.</AI>

이 결정은 `성공 결과 자동 반영`이 아니다. 결과 생성만으로는 아무것도 바꾸지 않으며, 사용자가 본문·캐논 후보를 본 뒤 누르는 명시적 승인 한 번을 안전한 최종 커밋으로 만든다.

## 수용 기준

- 잡 시작 때 전체 PLAY 원문 스냅샷이 생성 영수증에 저장되고 폴링 병합·새로고침 뒤에도 유지된다.
- `failed`·`timed-out`·`expired`·`cancelled` 중 recovery가 있는 항목만 두 복구 버튼을 렌더한다.
- TXT 본문과 파일명이 결정론적으로 생성되며 장면·역할 라벨·원문을 보존한다.
- WRITE 진입 전후 프로젝트의 회차 수·currentEpisode·캐논·인물·성장·품질/출간 입력이 동일하고, 작업본 본문은 빈 값이며 PLAY 원문은 별도 source에만 있다.
- 같은 영수증을 다시 열거나 새로고침해도 기존 작업 제목·본문이 유지된다.
- 명시적 `회차로 저장`은 빈 본문을 차단하고, 작성 본문만 원래 프로젝트의 편집 가능한 회차 하나로 추가하며 복구 메타데이터·원문 경고·캐논·인물 상태를 섞지 않는다.
- 본편 미반영 PLAY가 있으면 작업본 열기는 허용하되 회차 저장을 차단하고 작업본을 보존한다. 없으면 기존 PLAY session을 보존한 채 working project도 같은 새 회차로 동기화한다.
- localStorage 용량 실패가 서버 잡 시작 실패로 전파되지 않으며, 최신 미복구 원문을 우선 보존한다.
- 저장 실패가 이어지는 동안 기존·신규 미영속 영수증의 TXT 구제 표식을 모두 유지하고, 실제 저장 성공 때만 해제한다.
- 작업본 열기 뒤 영수증은 draft link를 유지하고 재열기를 허용하며, 명시 저장 뒤에만 `recoveredAt`을 기록하고 중복 저장 UI를 비활성화한다.
- 회차·working·receipt·draft 정리 중 어느 단계가 실패해도 영속 `commitIntent`로 재개하며, 이미 저장된 회차를 정확히 식별해 다음 회차를 중복 생성하지 않는다.
- legacy 환원은 receipt 저장 실패 뒤에도 `legacyRepair` journal로 다음 시작에서 완료되며, 작업본이 전역 보관함에서 고립되지 않는다.
- 잡 등록 전 실패 작업본도 로컬 생성 영수증을 가져 닫기·새로고침 뒤 생성 보관함에서 재열 수 있다.
- draft 저장만 성공하고 local receipt 저장이 실패한 경우 작업본을 닫지 않으며, receipt 영속을 확인한 뒤에만 보관함 재열기 가능 상태로 전환한다.
- 영수증의 `recoveryDraftId`가 있으면 해당 작업본을 우선 재열어 local receipt id 재해시로 빈 복제 작업본을 만들지 않는다.
- 여러 탭의 오래된 draft가 최신 `commitIntent`·`legacyRepair`를 지우지 않고, 이미 저장된 회차를 재사용해 중복 회차 생성을 막는다.
- 20개 정리는 빈 비활성 작업본부터 적용하며 active·작성 본문·부분 저장 journal은 조용히 삭제하지 않는다.
- 영수증의 openedAt/id와 recoveredAt/chapterId는 각각 완전한 쌍일 때만 유효하다.
- 이전 구현이 만든 회차는 엄격한 시스템 생성 조건을 모두 만족할 때만 작업본으로 환원되며, 편집·잠금·후속 회차가 있는 사용자 데이터는 그대로 남는다.
- 잡 등록 실패에서도 현재 PLAY 원문을 TXT/WRITE로 구제할 수 있다.
- 프로젝트 카드는 `작업 계속하기`로 표시한다. 0화·회차 없음·저장된 PLAY·작성 복구본 없음이 모두 맞을 때만 PLAY를 재개하며, 나머지는 WRITE로 폴백한다.
- 실패 구제 CTA와 작업실은 응결 결과가 아닌 수동 작성임을 명시하고, PLAY 현장 재시도는 사용자 클릭으로만 새 잡을 시작한다.
- PLAY 현장은 배열 위치가 아니라 `createdAt` 기준 현재 회차의 최신 생성 시도만 반영해 재시도 실행·성공 뒤 옛 실패 카드를 되살리지 않고, 지난 회차 실패를 현재 응결로 오인하지 않는다.
- 복구 가능 실패뿐 아니라 성공 영수증도 `localPersistenceFailed`이면 안전 문구 대신 새로고침 전 결과 검토·TXT 경고를 표시한다.
- 성공 결과 승인은 후보를 먼저 최신 본편과 대조하고, 충돌이 없을 때 PLAY working·본편을 같은 회차로 영속한 뒤 WRITE에 표시한다.
- 충돌 승인 후보는 기존 retcon/keep 검토를 통과하기 전 영속·영수증 제거를 하지 않으며, 검토 취소 뒤 응결 결과를 다시 볼 수 있다.
- 실제 resolved 회차·retcon checkpoint를 본편보다 먼저 영속하고, PLAY/WRITE 저장·빈 작업본 비활성화·영수증 제거 어느 단계가 실패해도 exact checkpoint로 재개한다.
- checkpoint 재개는 현재 본편이 저장된 직전/직후 상태 지문 중 하나와 정확히 맞을 때만 허용한다. 이후 WRITE·캐논 변경은 stale로 차단해 과거 승인이 새 편집을 우회하지 못하게 한다.
- checkpoint 이후 이어진 PLAY 대화는 다시 N-2 방식으로 자르지 않는다. 최초 승인 때 저장한 turn 경계까지만 제거해 새 대화를 다음 응결 재료로 보존한다.
- 최초 승인도 승인 시점에 응결 범위를 다시 계산하지 않는다. 생성 시작 recovery 영수증에 실제 payload의 마지막 turn을 기록하고, 결과를 보류한 사이 이어진 대화는 그대로 남긴다. 경계가 없는 구버전 영수증은 0으로 취급해 대화 삭제보다 중복 보존을 택한다.
- poll·checkpoint·receipt 정리는 mutation 직전 durable inbox를 다시 읽어 대상 ID만 합치며, 실행 중 잡과 미완료 checkpoint는 일반 cap을 넘어도 유실하지 않는다.
- 승인 직전과 저장 직전에는 전체 PLAY session·working project를 exact 비교한다. 같은 작품에서 다른 탭이 새 대화를 추가했으면 stale 승인을 중단하고, 최신 session을 오래된 snapshot으로 덮지 않는다.
- running poll과 404 만료는 mutation 시점의 최신 영수증이 여전히 running일 때만 적용한다. 이미 succeeded/checkpoint가 된 영수증은 상태·결과·checkpoint를 그대로 보존한다.
- 유효한 늦은 succeeded 결과는 failed·cancelled·timed-out·expired보다 우선해 복구한다. checkpoint 뒤 부분 저장이 실패하면 durable PLAY를 다시 mount해 같은 화면의 재승인이 exact checkpoint로 이어진다.
- Chromium 로컬 앱은 single-writer Web Lock을 가진 탭 하나에서만 App·poller를 실행한다. 대기 탭은 localStorage를 쓰지 않으며 writer 종료 뒤 자동 승계한다.
- read-back은 exact 회차와 새 캐논뿐 아니라 승인 retcon statement를 PLAY working·본편 양쪽에서 확인한 뒤에만 영수증을 제거한다.
- 제목·본문·journal이 빈 복구 작업본은 성공 승인 뒤 삭제하지 않고 안전하게 비활성화해 일반 WRITE를 가리지 않는다.
- 전체 `bash init.sh`와 브라우저 실사용 흐름이 녹색이며 콘솔 오류가 없다.
