# P0-b 응결 실패 구제 — PLAY 기록을 잃지 않는 복구 경로 (2026-07-16)

## 문제

P0-a는 응결을 잡+폴링으로 분리해 화면 이탈에는 강해졌지만, 잡 시작 실패·생성 실패·취소·시간 초과·서버 재시작 만료 뒤에는 `폐기` 외의 다음 행동이 없다. 실제 플레이테스트에서는 긴 PLAY를 마친 사용자가 결과 생성에 실패한 뒤 대화 원문까지 잃었다고 느꼈다. P0-b의 목적은 생성 성공률을 다시 다루는 것이 아니라 **생성에 실패해도 사용자가 만든 PLAY 기록을 즉시 가져가거나 WRITE에서 계속 편집하게 하는 것**이다.

## Brainstorm — 설계 결정 질문과 답

### 1. 무엇을 복구 단위로 볼 것인가?

- 응결용으로 잘라낸 span이 아니라 사용자가 잡을 시작한 순간의 PLAY 전체 기록이 복구 단위다.
- `<AI>장면·작품명·예정 회차·전체 대화 원문을 하나의 `PlayRecoverySnapshot`으로 잡 시작 전에 캡처한다. 이후 PLAY가 바뀌어도 실패 영수증은 당시 원문을 보존한다.</AI>`

### 2. 어느 실패에서 구제 행동을 보여 줄 것인가?

- 잡 등록 자체가 실패하면 현재 PLAY 안에서 바로 보여 준다.
- 생성 보관함에서는 `failed`·`timed-out`·`expired`·`cancelled` 영수증에 보여 준다.
- `<AI>취소도 생성 결과는 없지만 사용자가 만든 원문은 다시 쓸 가치가 있으므로 같은 복구 경로를 제공한다. 성공 영수증은 기존 정밀 검토·승인 경로를 유지한다.</AI>`

### 3. 어떤 결과물로 구제할 것인가?

- `PLAY 기록 TXT` — 로컬 텍스트 파일로 즉시 내려받는다.
- `WRITE 초안으로 보내기` — 원래 `projectId`의 다음 회차에 편집 가능한 초안을 만들고 WRITE로 이동한다.
- `<AI>TXT에는 작품·회차·장면·보존 시각·역할 라벨이 붙은 원문을 담는다. WRITE 초안은 원문을 재창작하거나 요약하지 않고 그대로 옮긴다.</AI>`

### 4. 캐논과 메모리는 어떻게 보호할 것인가?

- 복구는 생성 결과 승인도, 캐논 승인도 아니다.
- WRITE 복구 회차의 `newCanonFacts`는 항상 빈 배열이며 잠그지 않는다.
- `<AI>`chapterFromDraftPayload`에 빈 intent·pressure를 전달해 성장 상태를 자동 갱신하지 않고, 기존 캐논 배열과 인물 상태가 그대로인지 순수 테스트로 고정한다.</AI>`
- 앵커 위반 턴도 원문 TXT/초안에는 남는다. 다만 캐논 후보로 승격하지 않는다. 원문 보존과 캐논 승인을 분리한다.

### 5. 중복 전송과 이전 데이터는 어떻게 다룰 것인가?

- `<AI>생성 보관함 영수증에 `recoveredAt`과 `recoveredChapterId`를 기록해 한 번 WRITE로 보낸 항목은 `WRITE로 보냄` 비활성 상태로 바꾼다. 같은 스냅샷의 포맷된 원문이 이미 회차에 있으면 도메인에서도 재사용해 이중 클릭을 막는다.</AI>`
- `<AI>구버전 영수증은 recovery 필드 없이 그대로 파싱한다. 복구 스냅샷이 없는 오래된 영수증에는 불가능한 버튼을 만들지 않는다.</AI>`
- 원래 작품이 보관함에 없으면 다른 작품에 넣지 않고 TXT 보관을 안내한다.

### 6. 본편과 PLAY 작업본이 갈라져 있으면 어떻게 할 것인가?

- `<AI>PLAY에 본편 미반영 회차·캐논이 있으면 WRITE 복구를 자동 병합하지 않고 ⟳최신화를 먼저 요구한다. 캐논 승인 게이트를 복구 편의로 우회하지 않기 위해서다.</AI>`
- `<AI>미반영 항목이 없으면 복구 회차를 본편에 저장한 뒤 기존 PLAY session은 보존하고 작업본 project만 같은 본편 스냅샷으로 맞춘다. 복구 뒤 PLAY를 계속해도 다음 회차 id가 충돌하지 않는다.</AI>`

### 7. 로컬 보관공간이 부족하면 어떻게 할 것인가?

- `<AI>영수증 저장 실패는 이미 시작된 서버 잡의 실패로 간주하지 않는다. 성공 결과·실행 중 잡·미복구 PLAY 원문은 삭제하지 않고, 성공 결과에 중복으로 붙은 복구 snapshot만 줄여 한 번 재시도한다.</AI>`
- `<AI>재시도도 실패하면 메모리 UI에는 기록을 유지하고, 실행 중 상태라도 새로고침 전 TXT 보관 행동과 경고를 함께 표시한다.</AI>`

## 사용자 계약

1. 응결을 시작하는 순간 당시 PLAY 전체 기록을 로컬 생성 영수증에 함께 보존한다.
2. 생성 실패·취소·시간 초과·연결 만료 뒤에도 원문 TXT와 WRITE 초안 두 구제 행동을 제공한다.
3. 잡 등록 자체가 실패해도 현재 PLAY 안에서 같은 두 행동을 바로 제공한다.
4. TXT는 생성된 문장이 아닌 사용자의 PLAY 원문이며, 파일명은 작품명과 예정 회차를 식별할 수 있다.
5. WRITE 전송은 원래 작품만 활성화한 뒤 편집 가능한 새 회차를 만든다. 다른 작품으로 조용히 우회하지 않는다.
6. WRITE 복구는 캐논·인물 상태를 자동 변경하지 않는다. 사용자가 편집하고 기존 잠금/검토 흐름을 거친다.
7. 한 영수증에서 WRITE 전송을 완료하면 재전송 버튼을 비활성화하되 TXT 내려받기는 계속 허용한다.
8. 본편 미반영 PLAY가 있으면 먼저 최신화를 요구하며, 안전할 때는 복구 뒤 본편과 PLAY 작업본의 다음 회차를 일치시킨다.

## 데이터 모델

```ts
interface PlayRecoverySnapshot {
  schema: 'storyx/play-recovery/v1';
  projectId: string;
  projectTitle: string;
  episode: number;
  scene: string;
  transcript: string;
  capturedAt: string;
}

interface GenerationInboxItem {
  // 기존 필드 유지
  recovery?: PlayRecoverySnapshot;
  recoveredAt?: string;
  recoveredChapterId?: string;
  localPersistenceFailed?: boolean; // 메모리 전용, localStorage에는 기록하지 않음
}
```

스냅샷은 기존 생성 보관함 localStorage에만 보존한다. 서버 잡 요청과 API 응답 스키마에는 원문 복구 필드를 추가하지 않는다. 구버전 데이터와 서버 계약을 건드리지 않는 클라이언트 구제 계층이다.

## UI

### ProjectHub 생성 보관함

- 실패 계열 영수증 아래에 `PLAY 기록은 안전합니다. 원문 그대로 받거나 WRITE에서 이어갈 수 있어요.`를 표시한다.
- 행동 순서는 `PLAY 기록 TXT` → `WRITE 초안으로 보내기` → `폐기`다.
- WRITE 전송 뒤에는 두 번째 버튼이 `WRITE로 보냄`으로 바뀌고 disabled가 된다.
- 기존 `--nx-*` 라이트 브리지 토큰을 유지한다. 실패색은 상태 라벨에만 쓰고 복구 행동은 차분한 기본/primary 위계를 쓴다.

### PLAY 잡 등록 실패

- 기존 한 줄 오류 아래에 작은 복구 카드를 연다.
- 제목은 `응결은 멈췄지만 PLAY 기록은 안전합니다.`로 두고 같은 두 행동을 제공한다.
- 기존 `--st-*` 스튜디오 토큰만 사용한다. 새 애니메이션은 만들지 않는다.

## 비목표

- 생성 자동 재시도, 다른 모델 폴백, 서버 영속 큐.
- 성공 결과 자동 반영 또는 캐논 자동 승인.

## 수용 기준

- 잡 시작 때 전체 PLAY 원문 스냅샷이 생성 영수증에 저장되고 폴링 병합·새로고침 뒤에도 유지된다.
- `failed`·`timed-out`·`expired`·`cancelled` 중 recovery가 있는 항목만 두 복구 버튼을 렌더한다.
- TXT 본문과 파일명이 결정론적으로 생성되며 장면·역할 라벨·원문을 보존한다.
- WRITE 전송은 원래 프로젝트에 편집 가능한 회차 하나를 추가하고 캐논·인물 상태를 바꾸지 않는다.
- 본편 미반영 PLAY가 있으면 WRITE 전송을 차단하고, 없으면 기존 PLAY session을 보존한 채 작업본 project도 같은 회차로 동기화한다.
- localStorage 용량 실패가 서버 잡 시작 실패로 전파되지 않으며, 최신 미복구 원문을 우선 보존한다.
- 저장 실패가 이어지는 동안 기존·신규 미영속 영수증의 TXT 구제 표식을 모두 유지하고, 실제 저장 성공 때만 해제한다.
- WRITE 전송 후 영수증이 `recoveredAt`을 유지하고 중복 전송 UI가 비활성화된다.
- 잡 등록 실패에서도 현재 PLAY 원문을 TXT/WRITE로 구제할 수 있다.
- 전체 `bash init.sh`와 브라우저 실사용 흐름이 녹색이며 콘솔 오류가 없다.
