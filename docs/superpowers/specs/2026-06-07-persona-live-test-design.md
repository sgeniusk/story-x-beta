# 실사용 창작자 10인 풀 라이브 실증 테스트 — 설계 (Spec)

작성일 2026-06-07 · 상태 `draft` · 후속 `docs/superpowers/plans/2026-06-07-persona-live-test-plan.md`(예정)

## 0. 한 줄 정의

실제로 스토리X를 열 법한 한국 보통 사람 10명의 페르소나로, 앱을 **직접 조작**해 이야기를 만들고(장편은 **책 한 권 완권**까지 회차를 이어 쓰며) **반복 수정**하며, 그 전 과정을 면밀히 기록·평가해 새 제작 계획의 근거를 만든다.

## 1. 배경과 이번 테스트의 차별점

이전 평가는 두 번 있었다.
- `docs/story-x-12-creator-tester-report.md` (2026-05-14) — 창작 **전문가 유형** 합성 페르소나 12인이 **제품 방향**을 평가.
- `docs/story-x-20-expert-retest-report.md` (2026-05-15) — 전문가 20인 재테스트. 결론은 "방향은 맞다, 이제 기능이 아니라 한 작품을 끝까지 만드는 vertical slice로 증명하라".

이번 테스트는 결이 다르다.
- 평가자가 **전문가가 아니라 실사용 창작자**(웹소설 독자, 취미 글쓰기, 일상툰, 학부모 등)다.
- 추상 평가가 아니라 **실제 LLM 생성 + 반복 수정**을 앱에서 직접 돌린 **실증 데이터** 기반이다.
- 핵심 질문은 "방향이 맞나"가 아니라 **"보통 사람이 이 앱으로 한 편을 쓰고 10번 고쳐도 이야기가 안 무너지나"**다. 이는 스토리X의 북극성("이야기가 형태를 바꿔도 무너지지 않게")과 핵심 차별점("연속성을 제품 요건으로")을 정면으로 시험한다.

## 3. 테스트 프로토콜 (페르소나마다 동일, 풀 라이브)

dev 서버(`npm run dev`, 127.0.0.1:5173) + Playwright로 **앱을 직접 조작**한다. provider는 dev 기본값 **codex**(로컬). 편집기는 새 디자인 기본값 **floating**.

각 페르소나는 다음 사이클을 거친다.

1. **진입** — 랜딩 → 매체/포맷 선택 → 인터뷰(freewrite로 작품 아이디어 입력). 페르소나의 말투·기대로 입력한다.
2. **초안 생성** — 실 codex LLM. 첫 산출물을 받는다.
3. **검토** — 마진 5명 작가진(쇼러너·캐릭터·세계·장르·연속성)에게 전체 검토. 통과/수정/차단 의견 수집.
4. **~10회 수정 사이클** — 실제 창작자처럼 반복한다.
   - 검토 의견 반영(`acceptMarginDiff`)
   - 본문 직접 편집(빨간 표시로 남음)
   - 부분 재생성·이어쓰기
   - 재검토 → 다음 수정
   - 매 회차 "무엇을 고쳤고, 시스템이 무엇을 지켰고/망가뜨렸나"를 기록한다.
5. **(여력 시) 매체 변환 1회** — 같은 story core를 다른 형태로(예: 웹소설→웹툰 콘티). 변환 시 약속·문체·캐논이 보존되나 확인.

### 장편(#1~3) — 완권 연속 작성 (추가 규약)
장편은 위 사이클을 1화로 끝내지 않고, 책 한 권(~20~25화)을 향해 회차를 누적한다.
- 1화 완성 → 2화 → … → 완권까지. 매 회차는 직전까지의 캐논·캐릭터·설정을 컨텍스트로 이어받는다.
- **의도적 연속성 부하** — 중간(예: 5화·12화 시점)에 설정 변경·신규 캐릭터·관계 반전을 던져, 회차가 누적된 상태에서 캐논/캐릭터가 버티는지 시험한다.
- **캐릭터 추적표** — 주요 인물의 이름·말투·외형·관계·능력을 회차별로 대조해 드리프트를 잡는다(#3 다수 인물이 핵심 부하, #2 장기 감정선).
- 완권 후 전체 통독으로 캐릭터·캐논·문체 드리프트를 종합 평가한다.
- **멀티세션 전담** — 세션 경계는 회차 단위로 끊고 `session-handoff.md`에 "현재 N화·다음 회차가 이어받을 컨텍스트"를 남긴다.

### 기록물 (페르소나마다)
- 단계별 스크린샷 (`docs/reviews/2026-06-07-persona-live-test/<id>/`)
- 수정 이력 표 (회차 / 무엇을 고침 / 시스템 반응 / 연속성 영향)
- 콘솔 에러 로그
- 페르소나의 1인칭 소감(막힌 곳·좋았던 곳)

## 4. 평가 축 6개 + 판정

1점 약함 ~ 5점 강함.

1. **생성 품질** — 첫 초안이 그 분야 기준으로 쓸 만한가.
2. **수정 반응성** — 수정 지시를 정확히 반영하나, 엉뚱하게 바꾸진 않나.
3. **연속성 유지 ★** — 10회 수정 후에도 캐논·문체·설정이 안 무너지나. (스토리X 핵심 차별점의 실증)
4. **매체 적합성** — 그 분야 관습(웹소설 훅·컷 리듬·낭독성·주장-근거 등)에 맞나.
5. **UX 마찰** — 막힘·혼란·통제감. 보통 사람이 끝까지 가나.
6. **상업성** — 이 이야기가 시장에서 팔릴/발견될 가능성. 첫 문장·첫 컷·첫 30초·제목 경쟁력.

각 페르소나는 6축 점수 + 종합 **통과 / 수정 / 차단** 판정 + 가장 중요한 근거 1~2개를 남긴다. 차단 신호는 다수결로 무시하지 않는다.

## 5. 멀티세션 로드맵

전부 직접조작이라 한 세션에 못 끝낸다. 세션당 2~3명을 순차로 라이브 진행한다.

| 세션 | 범위 |
|---|---|
| S0 (오늘) | 이 spec 확정 + 실행 plan + **#1 파일럿**(도입 1~3화 풀 라이브 + 수정 + 회차 연속성 확인 → 프로토콜·기록 양식 검증, 1화당 소요 측정) |
| S1~S3 | #1 웹소설 장편 **완권 완주**(~20~25화 회차 누적) |
| S4~S6 | #2 로맨스판타지 장편 완권 |
| S7~S9 | #3 헌터물 장편 완권 (다수 캐릭터 일관성 집중) |
| S10 | #4 미스터리 중편 (캐릭터 일관성 집중) |
| S11 | #5·#6 단편 2종 |
| S12 | #7·#8 인스타툰·에세이 |
| S13 | #9·#10 동요읽기·논문 |
| S14 | **종합 실증 리포트** |
| S15 | 리포트 위에서 **새 제작 계획** 확정 |

장편 완권 세션 수(S1~S9)는 추정이다. S0 파일럿에서 측정한 1화당 소요로 plan에서 확정한다. 완권 풀라이브는 의도적으로 가장 무거운 경로이며, 총 **15+ 세션**이 든다. 세션마다 `session-handoff.md` 맨 위에 인계 노트를 남긴다.

## 7. 파일럿 (#1) 합격 기준

#1은 장편 완권이라 오늘 다 못 쓴다. 오늘 파일럿은 **장편 연속 작성 프로토콜의 도입부 검증**이다. 성공이라면 다음이 충족된다.
- dev + Playwright로 랜딩→인터뷰→1화 초안→검토→수정→**2~3화 누적**까지 **실제로** 돌았고, 회차 간 캐논·캐릭터가 이어지는지 확인했다.
- 기록 양식(스크린샷·수정 이력 표·캐릭터 추적표·6축 평가)이 한 페르소나분 틀로 완성됐다.
- **1화당 소요**(생성·검토·수정 시간)와 프로토콜 병목이 측정돼 S1~S9 완권 세션 수를 plan에서 확정할 수 있다.
