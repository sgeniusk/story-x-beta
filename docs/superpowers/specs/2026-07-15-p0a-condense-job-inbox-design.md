# P0-a 응결 신뢰성 — 로컬 Codex 잡 + 전역 생성 보관함 (2026-07-15)

## 문제

신규 사용자 실플레이에서 PLAY→회차 응결은 4/4 실패했다. Codex CLI는 약 141초 뒤 정상 결과를 만들었지만 브라우저 요청이 120초에 중단되어 결과가 버려졌다. 같은 재시도는 같은 실패를 반복했고, HTTP 연결과 자식 프로세스 수명이 결합되어 고아 프로세스 위험도 남았다.

현재 프로젝트 허브도 실제 다중 작품 저장소가 아니다. `serial-story-studio/project`와 `serial-story-studio/dive` 각 한 키만 사용하고, 허브는 현재 작품 카드 하나만 렌더한다. P0-a는 전체 작품 라이브러리를 만들지 않지만 생성 결과를 처음부터 `projectId`에 귀속시켜 후속 P0-c 다중 작품 라이브러리에서 재사용한다.

## brainstorm 확정 결정

1. 서버 프로세스 안의 메모리 잡 저장소를 사용한다. 브라우저 새로고침은 견디지만 Vite 서버 재시작 뒤 실행 중 잡 복구는 비목표다.
2. 같은 입력의 진행 중 잡은 재사용하고 `jobId`로 다시 연결한다. 중복 Codex 프로세스를 만들지 않는다.
3. 화면 이탈은 잡을 중단하지 않는다. 명시적 취소, 5분 실행 상한, Vite 종료 때만 자식 프로세스를 종료한다.
4. 완료 결과는 회차에 자동 반영하지 않는다. 전역 생성 보관함에 도착한 뒤 사용자가 작품에서 검토·승인해야 회차와 캐논이 바뀐다.
5. P0-a 보관함 UI는 프로젝트 화면 중심 + PLAY 바로가기다. 전역 플로팅 UI는 만들지 않는다.
6. 생성 실행기는 ChatGPT 계정으로 로그인된 로컬 Codex CLI다. 잡·보관함 계약은 향후 BYOK/Cloud 실행기로 교체 가능하게 provider 비종속으로 둔다.
7. 다중 작품 저장·임시작→정식작 전환은 바로 다음 P0-c 작품 라이브러리 코어로 분리한다.

## 상태와 데이터 계약

서버 잡 상태는 `running | succeeded | failed | cancelled | timed-out`이다. 잡에는 `id`, `kind`, `projectId`, `baseRevision`, `episode`, `createdAt`, `updatedAt`, 완료 결과 또는 경고가 있다. 동일 요청 dedupe는 서버가 입력 전체의 안정 해시로 판단하며 terminal 잡은 새 실행을 막지 않는다.

브라우저 전역 보관함은 최대 20개를 최신순으로 localStorage에 보존한다. 손상 항목은 드랍하고 결과 shape가 없는 성공 항목은 실패로 강등한다. 서버가 재시작되어 알려진 잡을 찾지 못하면 `expired`로 표시하되 이미 브라우저가 회수한 성공 결과는 유지한다.

`baseRevision`은 응결 시작 시점의 작품 핵심 상태를 나타내는 결정론적 문자열이다. 승인 시 현재 revision과 다르면 `기준이 오래됨` 경고를 보여 주고 기존 연속성 검토를 거친다. P0-a는 결과를 자동 폐기하지 않는다.

## API

- `POST /api/dive-condense-jobs` — 입력과 메타데이터를 받아 즉시 잡 스냅샷 반환.
- `GET /api/dive-condense-jobs/:id` — 상태와 완료 결과 반환.
- `DELETE /api/dive-condense-jobs/:id` — 실행 중 자식 종료 후 cancelled 반환.

일반 `storyxBridge` 요청은 응답 연결이 완료 전에 닫히면 자식 프로세스에 종료를 전파한다. 잡 실행은 제출 HTTP 수명과 분리되며 취소·상한·서버 종료만 종료 원인이다.

## UI 흐름

```text
PLAY에서 지금 응결
  → 잡 생성, PLAY는 계속 사용 가능
  → 프로젝트 허브 생성 보관함에 running 표시
  → 완료 시 succeeded + 회차 제목 표시
  → 작품에서 검토
  → 기존 응결 승인 카드에서 본문·캐논 후보 확인
  → 승인하면 회차 커밋 후 보관함 항목 제거
  → 보류하면 보관함 유지 / 폐기하면 제거
```

프로젝트 허브의 보관함은 기존 밝은 `pjx-*` 미학 안에서 작업 영수증처럼 보이게 한다. PLAY에는 진행 상태, 취소, 보관함 이동만 제공한다. 새 색상 체계를 만들지 않고 기존 `--nx-*`와 스튜디오 `--st-*` 토큰을 지킨다.

## 실패·보존 계약

- HTTP 제출 실패는 잡이 만들어지지 않은 것으로 표시하고 다시 시도 가능하다.
- 생성 실패·시간 초과·취소는 원문 PLAY와 프로젝트를 변경하지 않는다.
- 실패 경고는 정확한 상태를 표시하며 stderr 전문이나 민감 원고를 UI에 노출하지 않는다.
- 성공 결과도 사용자 승인 전 project·chapter·canon·memory bank를 변경하지 않는다.
- 보관함 원문에는 응결 결과만 저장하고 전체 프롬프트·대화 전문을 중복 저장하지 않는다.

- 서버 재시작 중 실행 복구와 디스크 잡 큐.
- 다중 작품 레지스트리, 임시작 승격, 검색·보관·완결 관리(P0-c).
- prod Vercel dive Function과 Story X Cloud 과금.
- 실패 시 트랜스크립트 반출(P0-b).

## 합격 기준

1. 120초 HTTP 연결 없이 141초급 로컬 응결이 서버에서 완주할 수 있다.
2. 새로고침·화면 이탈 뒤 알려진 `jobId`를 다시 폴링한다.
3. 같은 진행 입력은 한 Codex 자식만 실행한다.
4. 완료 결과가 프로젝트 허브 보관함에 나타나고 승인 전 작품·캐논 변화는 0이다.
5. 취소·5분 상한·Vite 종료가 자식 종료로 이어진다.
6. 기존 PLAY 대화·응결 승인·연속성 게이트가 약화되지 않는다.
