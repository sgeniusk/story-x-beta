## 0. 이 문서가 결정하는 것

이 문서는 Story X가 장기적으로 무엇이 될지와, 지금 누구의 어떤 문제를 먼저 풀지를 분리해 고정한다. 기능 수를 늘리는 로드맵이 아니라 다음 네 질문에 답하는 개발 기준이다.

1. Story X는 어떤 서비스인가.
2. 첫 번째 핵심 사용자는 누구인가.
3. 사용자가 어떤 결과를 얻어야 제품이 성공한 것인가.
4. 어떤 순서로 개발해야 차별점이 실제 가치가 되는가.

근거는 다음 정본과 실측 증거다.

- `docs/storyx-evaluator-development-update.md` — 북극성, P0 운영 구조, story-first 원칙.
- `docs/research/2026-06-28-dive-x-market-direction.md` — 대화→회차→캐논→다음 플레이 폐루프와 시장 포지션.
- `docs/decisions/2026-06-10-market-proof-1.0.md` — 외부 작가 3명·각 10화의 시장증명 게이트.
- `docs/reviews/2026-07-13-playtest-persona-review/` — 신규 사용자 실플레이와 5/5 차단 판정.
- `docs/handoff/2026-07-14-codex-handoff.md` — 현재 신뢰성 백로그와 Codex 실행 계약.

## 1. 본질적인 서비스 지향점

### 1.1 한 문장 정의

> **Story X는 사용자가 이야기 속에서 대화하고 선택한 경험을, 다음 사람이 읽을 수 있는 한국어 연재 회차로 응결하고, 그 결과를 캐논으로 기억해 다음 화까지 이어 주는 개인 연재 스튜디오다.**

### 1.2 무엇이 제품이고 무엇이 수단인가

| 구분 | Story X에서의 의미 |
|---|---|
| 입력 방식 | PLAY 대화, 자유 서술, 함께 구상, 적응형 인터뷰 |
| 핵심 산출물 | 읽히고 수정 가능하며 반출 가능한 한국어 연재 회차 |
| 반복 가치 | 승인된 캐논·관계·목소리·미회수 약속이 다음 화에 이어짐 |
| 품질 장치 | 독립 작가진 검토, 연속성 계약, 문체·의외성·페이오프 게이트 |
| 장기 확장 | 같은 이야기 코어를 웹툰·오디오북·영상으로 옮기는 매체 브리지 |

따라서 **챗은 제품 자체가 아니라 창작 진입 방식**이고, **다매체는 첫 제품이 아니라 검증된 이야기 코어의 후속 확장**이다. 지금의 제품 성공 여부는 에이전트 수나 지원 매체 수가 아니라, PLAY가 회차로 남고 다음 화가 더 쉬워지는지로 판정한다.

### 1.3 사용자에게 주는 핵심 약속

1. **놀고 끝나지 않는다.** 모든 PLAY 세션은 응결 회차 또는 최소한 회수 가능한 텍스트로 남는다.
2. **내 이야기의 주도권은 사용자에게 있다.** AI가 만든 새 사실은 승인 전 캐논이 되지 않는다.
3. **길어져도 같은 작품으로 남는다.** 인물 욕망, 관계 온도, 세계 규칙, 미회수 약속과 문체가 다음 화에 이어진다.
4. **좋은 이야기를 위해 필요한 마찰은 숨기지 않는다.** 충돌과 대가는 드러내되, 내부 엔진 용어와 사무적 마찰은 줄인다.
5. **내가 만든 결과물을 들고 나갈 수 있다.** 복사·TXT·표준 포맷 반출은 부가기능이 아니라 소유권의 표현이다.

### 1.4 지켜야 할 제품 원칙

- **Story first, medium second.** 먼저 읽히는 이야기와 회차를 증명한 뒤 매체를 확장한다.
- **PLAY first, artifact always.** 대화로 시작하되 매 세션은 산출물로 끝난다.
- **User approval before memory.** 승인되지 않은 사실·수정·생성물은 캐논과 메모리 뱅크로 새지 않는다.
- **Continuity is a product requirement.** 연속성 검사는 생성물을 예쁘게 보이게 하려고 약화하지 않는다.
- **Preserve artistic friction, remove operational friction.** 작품의 어려운 결정은 보여주고, 대기·복구·반출·용어 해독의 어려움은 제거한다.
- **Evidence before breadth.** 새 매체·새 에이전트보다 실제 완성 회차, 재방문, 반출, 10화 완주 증거를 우선한다.

## 2. 대상 목표

### 2.1 1차 핵심 사용자 — 한국어 장르 연재 창작자

첫 시장은 **한국어 웹소설을 쓰고 싶거나 이미 연재 중인 개인 창작자**다. 장르는 현재 PLAY 품질과 실측 근거가 강한 로맨스판타지·헌터/회귀·미스터리/스릴러를 우선한다.

핵심 상황은 다음과 같다.

- 머릿속 소재나 캐릭터는 있지만 빈 문서에서 첫 장면을 시작하기 어렵다.
- 퇴근 후 30~90분처럼 짧은 시간에 한 회차의 재료와 초안을 얻고 싶다.
- 주 3~5회 연재를 위해 회차당 약 5천자 수준의 산출물이 필요하다.
- 여러 화를 쓰면서 인물·설정·떡밥·문체가 흔들리는 문제를 겪는다.
- AI가 대신 쓴 글보다, 자신이 선택하고 고친 이야기를 원한다.
- 완성한 본문을 문피아·노벨피아·카카오페이지 계열 워크플로우로 쉽게 옮기고 싶다.

### 2.2 1차 사용자 안의 두 진입 모드

| 모드 | 사용자의 출발점 | Story X가 제공할 경험 |
|---|---|---|
| **PLAY 입문자** | 작법은 모르지만 캐릭터와 장면을 놀아 보고 싶음 | 가벼운 설정→상대와 PLAY→읽히는 첫 회차→쉬운 말의 다음 행동 |
| **연재 실무자** | 마감, 분량, 설정 관리, 반출이 중요함 | 빠른 회차화→5천자급 초안→수정/검토→캐논 반영→TXT/복사 |

두 모드는 별도 제품이 아니다. 같은 폐루프를 사용하되 입문자는 전문 용어를 숨긴 안내형 표면을, 실무자는 분량·캐논·게이트를 직접 다루는 고급 표면을 사용한다.

### 2.3 2차 사용자

- 장편 에세이·연재 에세이 작성자: lived material 인터뷰와 목소리 보존이 핵심인 사용자.
- 기존 장편 작가: 이미 쓴 원고의 캐논·관계·미회수 약속을 관리하려는 사용자.
- 웹툰·오디오 제작자: 텍스트 코어가 충분히 검증된 뒤 매체 변환 패킷을 사용하는 사용자.

2차 사용자는 아키텍처에서 배제하지 않지만, 1차 폐루프의 신뢰성과 완성도를 희생해 우선 구현하지 않는다.

### 2.4 현재의 비타깃

- 무제한 캐릭터챗·연애 컴패니언 자체를 원하는 사용자.
- 이미지·영상 생성이 첫 번째 목적인 사용자.
- 팀 협업·출판사 권한 관리가 먼저 필요한 엔터프라이즈 고객.
- 버튼 한 번으로 무검수 완성작을 원하는 사용자.
- 모든 매체를 동시에 시작하려는 제작 파이프라인.

이 경계는 Story X가 캐릭터챗의 무료·즉시성 경쟁이나 범용 생성기의 기능 수 경쟁으로 끌려가는 것을 막는다.

## 3. 핵심 사용자 여정과 제품 계약

### 3.1 핵심 폐루프

```text
소재 발견
  → 가벼운 설정 확인
  → PLAY로 장면과 선택 축적
  → 한 회차로 응결
  → WRITE에서 사용자 수정
  → 검토·충돌 확인
  → 사용자 승인
  → 캐논·관계·목소리 갱신
  → 복사/TXT 반출
  → 갱신된 기억으로 다음 PLAY
```

### 3.2 반드시 지켜야 할 서비스 계약

| 계약 | 실패로 보는 상태 | 통과 기준 |
|---|---|---|
| 세션 보존 | 생성 실패 후 사용자가 얻는 것이 없음 | 응결 성공 또는 트랜스크립트/초안 구제 경로 제공 |
| 기다림의 가시성 | 수십 초 동안 정적 문구·검은 공백 | 경과 시간, 단계, 취소/복구 상태가 보임 |
| 회차 산출 | 1,500~3,000자 고정, 수정·반출 어려움 | 목표 분량 선택, WRITE 반영, 복사/TXT 가능 |
| 창작 주도권 | 선택지 즉시 전송, 새 사실 자동 캐논화 | 편집 후 전송, 변경 미리보기, 사용자 승인 |
| 장편 연속성 | 앞선 규칙·관계·목소리가 다음 화에서 소실 | 충돌은 차단하고 승인된 변화만 다음 컨텍스트에 주입 |
| 복귀 가능성 | 재접속 시 빈 WRITE로 착지해 유실로 오인 | 마지막 활성 작품·모드·진행 작업으로 복귀 |

### 3.3 2026-07-14 라이브 UI 관찰

로컬 서비스의 실제 랜딩→온보딩→PLAY→WRITE→PLAN→작가실→캐논 동선을 직접 확인했다. 현재 화면은 **Story X의 철학을 소개하는 데는 성공했지만, 사용자가 작가진과 캐논을 운용하는 서비스 UX는 아직 약하다.**

| 표면 | 현재 강점 | 현재 핵심 문제 | 목표 변화 |
|---|---|---|---|
| 랜딩 | “한국어 장편 연재”, “작가 승인”, “같은 이야기”가 명확함 | 에디터 미리보기의 복잡도와 첫 실제 PLAY 경험 사이 간극이 큼 | 첫 CTA 뒤 3분 안에 PLAY의 재미와 회차 산출 계약을 체험 |
| 온보딩 | 매체·형식 카드와 단계 구조가 읽힘 | 첫 시장이 소설 연재인데 5개 매체가 같은 무게로 보여 선택 비용이 큼 | 소설 장편을 추천 경로로 두고 다른 매체는 확장 경로로 정리 |
| PLAY | 장면·상대·행동 입력에 집중할 수 있음 | 화면이 비어 보이고 `쇼러너`, `응결`, `전개`의 차이와 현재 담당자가 설명되지 않음 | 장면 진행, 현재 리드, 다음 행동, 축적된 회차 재료를 한 흐름으로 표시 |
| WRITE | 중앙 원고 집중 표면이 분명함 | 빈 초안에서 PLAY 재료·활성 캐논·검토 필요 이유가 끊김 | PLAY에서 온 재료, 원고, 필요한 검토와 반출을 같은 회차 맥락으로 연결 |
| PLAN | 작품 데이터 도구의 범위는 넓음 | `하니스`, `온톨로지`, 강제 실패 수치가 창작자의 다음 행동으로 번역되지 않음 | “무엇이 막혔고 무엇을 고치면 되는지”를 먼저, 엔진 지표는 상세 보기로 이동 |
| 작가실 | 6개 역할과 전문 분야가 존재함 | 6명이 동등하게 대기하고 “전체 검토”만 강조되어 누가 왜 필요한지 알 수 없음 | 한 명의 리드와 필요한 전문가만 호출하고, 선택 이유·읽은 기억·영향을 표시 |
| 캐논 | 인물·장소·사물·사건·시간선 구분이 있음 | 개수만 보여 승인 대기, 확정 사실, 충돌, 변경 영향의 생애주기가 보이지 않음 | 후보→영향 확인→승인/수정/거절→확정의 의사결정 화면으로 전환 |

### 3.4 UI/UX 북극성

> **사용자는 에이전트 조직도를 조작하는 사람이 아니라, 자기 회차를 만드는 작가다. 시스템은 매 순간 필요한 한 명을 앞세우고, 캐논에 영향이 생길 때만 결정권을 요청해야 한다.**

이를 위해 스튜디오 UI는 다음 4계층으로 단순화한다.

1. **중앙 작업 표면** — PLAY의 장면, WRITE의 원고, PLAN의 회차·캐논처럼 지금 해야 할 창작 행위 하나만 중심에 둔다.
2. **다음 행동 바** — `장면 계속하기`, `한 화로 만들기`, `원고 고치기`, `변경 승인하기`, `TXT로 내보내기`처럼 결과 중심 동사를 쓴다.
3. **작가진 상태** — 기본은 `현재 리드 1명 + 필요한 전문가 최대 2명 + 연속성 게이트`만 보인다. 전체 조직도는 상세 보기다.
4. **변경 승인함** — 캐논·관계·목소리 변경이 있을 때만 나타나며, 변경 전/후와 영향 회차를 보여 준다.

Beginner 표면에서는 `응결`, `하니스`, `온톨로지`, 에이전트 ID를 숨기고 `한 화 만들기`, `작품 건강`, `설정 관계도`로 번역한다. Pro 표면은 같은 판정과 데이터를 유지하되 목표 글자 수, 캐논 층, 게이트 근거와 컨텍스트 패킷을 펼쳐 볼 수 있게 한다.

### 3.5 에이전트 선택 원칙

에이전트가 스스로 무리를 이루거나 LLM이 임의로 다음 에이전트를 고르게 하지 않는다. **결정론적 라우터가 작업 맥락으로 작가진을 선택하고, LLM은 선택된 역할 안에서만 생성·검토한다.**

라우터 입력은 다음 5개다.

- `stage`: 온보딩 / PLAY / WRITE / PLAN / 승인 / 출판
- `medium`: 소설 / 에세이 / 만화 / 오디오북
- `intent`: 소재 찾기 / 장면 진행 / 회차화 / 문장 수정 / 구조 수정 / 캐논 변경 / 반출
- `artifactState`: 빈 상태 / PLAY 축적 중 / 초안 있음 / 검토 실패 / 승인 대기 / 출간 준비
- `riskFlags`: 인물 정체성 / 세계 규칙 / 시간선 / 관계 / 문체 / 개인정보 / 매체 변환 영향

라우터 출력은 항상 다음 형식이다.

```text
Lead 1명
  + Specialist 0~2명
  + Gatekeeper 0~1명
  + 각 에이전트용 최소 Context Packet
  + 사용자에게 보일 선택 이유
```

호출 강도는 세 단계로 제한한다.

| 강도 | 자동/수동 | 구성 | 사용 시점 |
|---|---|---|---|
| 빠른 도움 | 자동 | Lead 1명 | PLAY 다음 행동, 문장 한 구절, 막힘 해소 |
| 표준 작업 | 자동 | Lead + Specialist 1~2명 + 필요 시 Gatekeeper | 회차 응결, 구조 변경, 캐논 후보 생성 |
| 깊은 검토 | 사용자 명시 | 독립 검토 최대 5명 후 종합 | 완성 회차, 출간 전, 큰 설정 변경 |

“6명에게 전체 검토”는 기본 CTA가 아니라 깊은 검토의 상세 옵션으로 내린다. 에이전트 수는 품질의 대리 지표가 아니며, 캐논 충돌은 어떤 검토 강도에서도 다수결로 통과시키지 않는다.

### 3.6 단계·행동별 기본 라우팅

| 사용자 행동 | Lead | 필요 시 Specialist | Gatekeeper | 선택 기준 |
|---|---|---|---|---|
| 첫 소재를 잡음 | Creative Coach | 소설은 Showrunner, 에세이는 Essay Interviewer | 없음 | 입력이 빈 상태인지 lived material인지 판별 |
| PLAY 장면을 진행 | Character Custodian | Showrunner, World Keeper | Continuity Editor는 위험 신호 때만 | 상대 인물의 욕망·관계가 중심, 세계 규칙 변경 시 세계 역할 추가 |
| PLAY를 한 회차로 만듦 | Showrunner | Genre Stylist, Voice Curator | Continuity Editor | 회차 약속·클리프행어를 잡고 문체를 적용한 뒤 모순 차단 |
| WRITE에서 선택 구절 수정 | Voice Curator | Character Custodian 또는 Genre Stylist | 위험 신호 때 Continuity Editor | 선택 영역과 수정 의도에 맞춰 최소 역할만 호출 |
| PLAN에서 큰 구조 변경 | Showrunner | Timeline Keeper, Bible Curator | Continuity Editor | 영향 회차와 미회수 약속을 먼저 계산 |
| 인물·세계 설정 변경 | Canon Librarian | Character Custodian 또는 World Keeper, Timeline Keeper | Continuity Editor | 변경 종류로 전문 역할을 고르고 승인 전 영향 미리보기 |
| 캐논 후보 승인 | Canon Librarian | Memory Evolution Keeper | Continuity Editor | Hard/Living/Soft 분류와 충돌 해소 뒤에만 영속화 |
| 에세이 회차화 | Essay Curator | Essay Interviewer, Voice Curator | 개인정보·사실성 규칙 | 사용자의 실제 경험을 발명하지 않고 노출 윤리와 목소리 보호 |
| 출간·반출 | Publishing Distribution Manager | Work Library Manager, Platform Curator | 품질 게이트 | 매체·플랫폼 포맷과 승인 상태에 따라 패키징 |

각 실행은 UI에 `현재 담당`, `선택된 이유`, `읽은 기억`, `제안`, `캐논 영향` 다섯 항목으로 추적 가능해야 한다. 기본 화면에는 한 줄 요약만 보이고, 사용자가 펼치면 근거를 확인할 수 있다.

### 3.7 캐논 유지 생애주기

캐논은 원고에서 자동 추출해 곧바로 저장하는 데이터가 아니다. 다음 승인 파이프라인을 따른다.

```text
PLAY/WRITE 결과
  → 새 사실 후보 추출
  → Hard / Living / Soft 분류
  → 기존 캐논·시간선·관계와 충돌 검사
  → 영향 회차와 다음 PLAY 변화 미리보기
  → 사용자 승인 / 문장 수정 / 거절
  → 승인된 항목만 canon·timeline·bible에 기록
  → 역할별 Context Packet 재생성
```

세 층의 의미는 사용자 언어로 다음처럼 표시한다.

| 내부 층 | 사용자 표시 | 규칙 |
|---|---|---|
| Hard | **고정 설정** | 이미 출간되었거나 작품 정체성을 이루는 사실. 변경은 영향 미리보기와 명시 승인 필수 |
| Living | **현재 상태** | 관계 온도, 부상, 위치, 소유물처럼 회차에 따라 변함. 이전 값과 변경 이력을 유지 |
| Soft | **가능성/떡밥** | 아직 확정되지 않은 암시·가설·후보. 다음 에이전트가 사실처럼 단정하지 못함 |

충돌은 `오류` 한 단어로 끝내지 않고 사용자가 세 가지 중 하나를 고르게 한다.

1. **반전으로 인정** — 모순처럼 보인 차이를 의도된 reveal로 승격하고 회수 약속을 만든다.
2. **이번 원고 수정** — 기존 캐논을 유지하고 현재 회차의 문장을 고친다.
3. **설정 변경** — 기존 캐논을 바꾸되 영향 회차와 후속 수정 목록을 만든다.

거절한 후보와 원문·인터뷰 같은 private 자료는 캐논 컨텍스트에 유입하지 않는다. 에이전트는 전체 원고가 아니라 `buildMemoryBankContextPacket` 계열의 역할별 최소 패킷만 읽으며, 패킷이 무엇을 포함했는지 사용자가 확인할 수 있어야 한다.

### 3.8 에이전트·캐논 UX의 구현 순서

| 순서 | 슬라이스 | 사용자에게 보이는 결과 | 선행 조건 |
|---|---|---|---|
| UX-0 | 용어·다음 행동 정리 | PLAY/PLAN의 내부 용어가 창작 행동으로 번역됨 | Phase 0 진행 피드백과 병행 가능 |
| ORCH-1 | 결정론적 Agent Router | 상황에 맞는 Lead와 선택 이유가 자동 표시됨 | 기존 `planAgentRuns`·context packet 계약 재사용 |
| ORCH-2 | 선택 영역·위험 신호 라우팅 | 필요한 전문가만 추가되고 전체 검토 남발이 사라짐 | ORCH-1 |
| CANON-1 | 승인함 + Hard/Living/Soft | 새 사실이 자동 저장되지 않고 한곳에서 결정됨 | P0 회차 보존, 기존 approval queue 재사용 |
| CANON-2 | 변경 영향 미리보기 | 설정 변경 전 영향 회차·관계·다음 PLAY 변화를 확인 | CANON-1, canon refactor 재사용 |
| MEMORY-1 | 패킷 투명성·재주입 | 누가 무엇을 읽었는지 보이고 승인된 기억이 다음 화에 이어짐 | ORCH-1, CANON-1 |

이 트랙은 Phase 0의 신뢰성보다 앞설 수 없다. 다만 UI 용어 정리와 라우터 명세는 병행 설계할 수 있고, 실제 자동 호출과 캐논 영속화는 응결·복구 계약이 안정된 뒤 연결한다.

## 4. 성공 목표와 지표

### 4.1 북극성 지표

> **주간 ‘반출 가능한 회차’ 완성 수**

단순 생성 수나 채팅 턴 수가 아니라, 사용자가 WRITE에서 확인하고 복사 또는 TXT로 반출할 수 있는 상태까지 도달한 회차를 센다. 이 지표는 PLAY의 재미, 응결 신뢰성, 수정 가능성, 반출 가치를 한 번에 묶는다.

### 4.2 단계별 제품 지표

| 단계 | 지표 | 초기 합격 기준 |
|---|---|---|
| 신뢰성 | 응결 요청 성공 또는 구제 산출물 확보율 | 내부 실측 100%, 유실 0건 |
| 활성화 | 신규 사용자가 첫 세션에 PLAY→회차→WRITE→반출 완료 | 외부 베타에서 측정, 실패 이유 전수 기록 |
| 체감 | 모든 장시간 LLM 작업의 진행 상태 가시성 | 100% 적용, 검은 공백 0건 |
| 산출 적합성 | 목표 분량과 실제 본문 분량의 일치 | 3천/5천/8천자 등 선택 범위별 허용 오차 정의 |
| 연재 가치 | 첫 회차 완료 사용자의 7일 내 두 번째 회차 완성 | 외부 베타 기준선 수집 후 목표 확정 |
| 품질 | 직접 캐논 모순 차단, 정상 재진술 오탐 | 기존 정밀도 기준선 유지, 약화 금지 |
| 시장증명 | 외부 작가 3명이 제작자 개입 없이 각 10화 이상 완성 | 3명 중 1명 이상 유료 지속 의향 또는 실과금 |

### 4.3 지금 세지 않을 허영 지표

- 총 생성 버튼 클릭 수.
- 총 에이전트 수.
- 지원 매체 수.
- 총 채팅 턴 수만으로 본 참여도.
- 랜딩 페이지 방문 수만으로 본 시장성.

이 값들은 회차 완성과 재방문으로 연결될 때만 보조 지표로 사용한다.

## 5. 개발 우선순위

### Phase 0 — 신뢰 바닥 복구

**목표:** 사용자가 투자한 플레이가 사라지거나 결과를 회수하지 못하는 경우를 없앤다.

1. **P0-a 응결 신뢰성**
   - 장시간 작업을 클라이언트 단발 타임아웃에 묶지 않는 실행 계약 확정.
   - 서버측 완주+결과 회수, SSE, 타임아웃+계측 중 하나를 brainstorm에서 선택.
   - 클라이언트 abort가 실제 작업 취소인지 결과 보존인지 명확히 하고 자식 프로세스 수명주기를 처리.
2. **P0-b 실패 구제 경로**
   - 응결 실패 위치에서 플레이 기록 TXT 다운로드와 초안 보내기 제공.
3. **P1-a 진행 피드백**
   - 대화·전개 후보·응결에 경과 시간과 단계 메시지 적용.
   - 응결 중 검은 공백과 강제 스크롤 수정.
4. **P1-b 텍스트 반출**
   - 회차 본문 원클릭 복사와 TXT 내보내기.

**종료 게이트:** 같은 실플레이 트랜스크립트로 응결 반복 성공, 실패 주입 시에도 텍스트 회수, 장시간 작업 전 구간 진행 상태 표시, 복사/TXT 실파일 확인.

### Phase 1 — 연재 가능한 한 회차 완성

**목표:** PLAY의 재미를 실제 연재 마감에 쓸 수 있는 산출물로 바꾼다.

1. 응결 분량을 고정 1,500~3,000자에서 목표 분량 계약으로 변경.
2. 3천/5천/8천자 또는 장르별 기본값을 brainstorm에서 결정.
3. 선택지 칩을 즉시 전송 대신 컴포저 삽입·편집 가능 방식으로 변경.
4. WRITE에서 글자 수, 회차 제목, 본문 수정과 저장 상태를 명확히 표시.
5. 마지막 활성 PLAY/WRITE 상태로 복귀하는 recovery 동선 정비.

**종료 게이트:** 신규 사용자가 첫 세션 안에 목표 분량 회차를 만들고, 한 문단 이상 직접 고친 뒤, TXT로 반출한다.

### Phase 2 — 쉬운 진입과 올바른 기대

**목표:** 작법을 모르는 사용자도 첫 재미까지 빠르게 도달하고, 고급 사용자는 불필요한 안내 없이 작업한다.

1. S3 적응형 인터뷰: 입력 유형을 먼저 파악하고 질문 수를 고정하지 않으며 충분해지면 종료.
2. 유사 프리셋은 정답 제시가 아니라 비교 앵커로만 사용.
3. 무관한 도윤·하란·세하 시드 칩 제거 또는 별도 ‘다른 시작점’으로 격리.
4. 건너뛴 인터뷰 완료 표시, 제목 절단, 빈 프로젝트 실패 지표 수정.
5. Beginner/Pro 표면 분리: `응결·하니스·온톨로지`보다 `한 화 만들기·설정 충돌·다음 행동`을 먼저 노출.

**종료 게이트:** 빈 상태 사용자 5명이 설명 없이 소재→PLAY 첫 응답까지 도달하고, 용어·진행 단계 오인으로 막히지 않는다.

### Phase 3 — 장편 연재 해자 강화

**목표:** 두 번째 화부터 Story X를 계속 써야 하는 이유를 만든다.

1. 승인된 캐논·관계 상태·목소리·미회수 약속을 다음 PLAY와 응결에 주입.
2. 회차별 캐논 delta와 승인 대기 항목을 쉬운 말로 표시.
3. 연속성뿐 아니라 페이오프 지연, 보상 루프, 관계 온도, 문체 drift를 추적.
4. 수정 전 영향 범위를 보여 주는 Refactor Impact Preview 연결.
5. 회차 목록, 다음 화 약속, 연재 리듬과 복귀 지점을 한 화면에서 관리.

**종료 게이트:** 한 작품 10화 연속 생성·수정에서 직접 모순 차단, 정상 변화 허용, 미회수 약속 추적, 사용자 승인 없는 캐논 유입 0건.

### Phase 4 — 배포 환경과 외부 시장증명

**목표:** 로컬 데모가 아니라 외부 사용자가 독립 완주하는 제품으로 전환한다.

1. P0-a에서 확정한 프로토콜로 prod `dive-*` Functions 구현.
2. dev/prod provider 계약, 오류·타임아웃·사용량 계측 통일.
3. 계정·클라우드 저장 또는 외부 베타에 충분한 안전한 동기화 경로 마련.
4. 연재 경험자 중심 외부 작가 3~5명 클로즈드 베타.
5. 각 참가자의 첫 세션, 첫 회차, 3화, 10화 이탈 원인을 구분해 기록.

**종료 게이트:** 외부 작가 3명이 제작자 개입 없이 각 10화 이상 완성하고, 최소 1명이 유료 지속 의향 또는 실과금 신호를 남긴다.

### Phase 5 — 매체 브리지 확장

**진입 조건:** Phase 4 시장증명 통과. 텍스트 폐루프의 신뢰성과 재방문이 확인되기 전에는 확장하지 않는다.

1. 에세이: lived material 인터뷰, 진실 계약, 목소리 보존.
2. 웹툰: beat→panel→bubble→scroll hook 패킷.
3. 오디오북: paragraph→speaker→pronunciation→pause→music/SFX cue 패킷.
4. 각 매체는 같은 Story Contract와 승인된 캐논을 공유하되, 매체별 편집 표면을 별도로 둔다.

**종료 게이트:** 같은 이야기 코어가 매체를 바꿔도 인물 욕망·세계 비용·관계 상태·핵심 질문을 보존하며, 각 매체의 실제 반출 포맷을 만든다.

| 응결 분량 파라미터 | Phase 1 | P0-a 완료 후 |
| 칩 컴포저 삽입 | Phase 1 | 독립 가능 |
| 시드 칩·인디케이터·제목 | Phase 2 | S3 분기 계약 보존 |
| 내부 용어·죄책감 UI | Phase 2 | 게이트 판정 자체는 변경 금지 |
| S3 적응형 인터뷰 | Phase 2 | Phase 0 완료 권장, S2 계약 유지 |
| prod `dive-*` Functions | Phase 4 | P0-a 실행 프로토콜 확정 후 |

이 순서의 핵심은 **“더 잘 시작하기”보다 “시작한 것을 반드시 회차로 남기기”가 먼저**라는 점이다.

## 7. 슬라이스 운영 방식

각 행은 하나의 feature 브랜치와 하나의 PR로 끝낸다.

1. **Brainstorm** — 사용자에게 제품 결정 질문을 제시하고 선택을 확정한다.
2. **Spec** — 사용자 계약, 비목표, 상태 전이, 실패/복구 경로, 파일 범위를 명시한다.
3. **Plan** — 테스트 선행 순서와 커밋 단위를 작성한다.
4. **TDD** — 실패 테스트를 먼저 추가하고 실제 로그를 근거로 구현한다.
5. **라이브 검증** — 로컬 실제 Codex 경로에서 사용자 여정을 끝까지 밟는다.
6. **종료** — `bash init.sh` 녹색, progress/handoff 갱신, 커밋, PR 생성. 머지는 사용자가 결정한다.

슬라이스마다 다음 질문에 답하지 못하면 착수하지 않는다.

- 어떤 사용자 손실을 없애는가.
- 사용자가 화면에서 어떤 변화를 체감하는가.
- 실패해도 무엇이 남는가.
- 기존 캐논·영속·프롬프트 미러 계약을 어떻게 보존하는가.
- 어떤 라이브 증거가 완료를 입증하는가.

## 8. 즉시 결정이 필요한 제품 쟁점

이 계획을 기준으로 첫 구현에 들어가기 전에 다음을 사용자와 확정한다.

1. **1차 쐐기 사용자 승인** — 한국어 장르 연재 창작자를 첫 핵심 사용자로 고정할지.
2. **P0-a 실행 모델** — 서버측 잡+폴링, SSE, 타임아웃+계측 중 어느 수준까지 첫 슬라이스에 포함할지.
3. **첫 회차 목표 분량** — 5천자 기본인지, 3천/5천/8천 선택인지.
4. **Beginner/Pro 분리 시점** — Phase 2에서 표면만 분리할지, 설정 모드까지 둘지.
5. **외부 베타 저장 방식** — BYOK+로컬 파일 동기화인지, 계정 기반 서버 저장을 먼저 둘지.

## 9. 계획의 최종 판정 문장

Story X는 “AI가 여러 매체를 만들어 주는 큰 시스템”부터 완성하지 않는다. 먼저 한 명의 한국어 연재 창작자가 캐릭터와 놀고, 그 시간을 읽히는 한 회차로 바꾸고, 자기 문장으로 고치고, 안전하게 들고 나가며, 다음 화에서도 같은 이야기로 이어지는 경험을 완성한다.

그 폐루프가 외부 작가의 10화 완주와 유료 지속 의향으로 증명된 뒤에야, Story X는 같은 이야기의 영혼을 웹툰·오디오·영상까지 옮기는 창작 운영체제로 확장한다.
