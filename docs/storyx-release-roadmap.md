# Story X Release Roadmap

> 기준: 2026-05-16 · `Alpha v0.8.0`. 베타까지는 로컬 CLI와 웹 에디터를 함께 쓰는 구조를 우선하고, 그 뒤 웹 서비스로 확장한다.
> 과거 버전별 변경 기록은 `docs/storyx-version-log.md`에 있다. 이 문서는 **앞으로의 계획**에 집중한다.

## 1. 지금 어디까지 왔나

`Alpha v0.8.0` 기준, 초안 생성과 작가진 검토가 실제 Claude 구독 호출로 동작한다. UI 골격과 핵심 도메인 계약은 테스트로 보호된다.

**되는 것**
- 랜딩 → 로그인 → 프로젝트 → 새 프로젝트 → 에디터 흐름, 편집/바이블 2트랙 에디터.
- 초안 생성·작가진 검토가 로컬 브리지를 거쳐 실제 Claude 구독으로 동작(L1~L3).
- 2화 이상 생성 시 확정 캐논·인물·세계 규칙을 LLM에 함께 넘겨 연속성 유지(L4).
- 메모리 승인 큐, 캐논/캐릭터/세계/문체 바이블 트랙, ⌘K 명령 팔레트.
- `storyx` CLI 하네스(doctor/draft/review/normalize), Vercel 자동 배포, `npm test` 113개 통과.

**아직 아닌 것**
- 검토에서 승인한 기억 후보가 실제 작품 캐논·바이블로 반영되는 루프가 닫히지 않았다.
- `storyx init`/`serve`/`memory sync` — 파일 기반 프로젝트가 없다.
- 장편 10화 이상 회귀 검증, 한국어 문체 유지 평가가 없다.
- 만화 컷 편집기·오디오 라인 편집기는 초기 설계 단계다.
- LLM 생성은 로컬 dev 브리지에서만 동작하고, 배포본은 기본 생성으로 폴백한다.
- 계정·결제·클라우드 저장·팀 협업은 미착수.

판정 — "로컬 알파". 다음 목표는 "기능을 더 보이게" 하는 것이 아니라 **생성–검토–승인–반영이 하나의 루프로 닫히게** 하는 것이다.

## 2. 제품 원칙

출시 기준은 "많이 생성한다"가 아니라 "오래 가도 무너지지 않는 이야기를 만든다"이다.

- 이야기가 먼저이고, 매체는 그 다음이다.
- 원고 전체를 매번 컨텍스트로 밀어 넣지 않고, 메모리 뱅크에서 역할별 컨텍스트 패킷을 만든다.
- 캐논 충돌은 다수결로 통과시키지 않는다. reveal / revision / blocked 중 하나로 분류한다.
- 사용자가 직접 고친 문장은 AI가 덮어쓰지 않는다. 검토·대안·영향 범위를 먼저 제시한다.
- 새 기억과 새 캐논은 사용자 승인 뒤에만 반영한다.
- 베타는 로컬 CLI에서 완성도를 증명하고, 이후 웹 서비스로 확장한다.

## 3. 다음 단계 — 집중 로드맵

### v0.9.0 · 승인 루프 닫기 (다음 작업)

검토가 만든 기억 후보가 승인을 거쳐 실제 작품 상태로 들어가는 고리를 완성한다. 이게 닫혀야 L2~L4가 의미를 갖는다.

- 검토 → `memoryCandidates` → 승인 대기함 → 사용자 승인 → `project`의 canon/character/world 갱신.
- 갱신된 캐논이 다음 회차 생성 컨텍스트(L4)에 자동 반영되는지 확인.
- 직접 편집 diff를 변경 블록 단위로 표시하고, 검토 요청 시 diff만 에이전트 패킷에 포함.
- `review-ledger`와 실패 로그를 기록으로 남긴다.

완료 기준 — 생성한 회차를 검토하고 후보를 승인하면, 그 사실이 바이블에 반영되고 다음 회차 생성이 그 캐논을 지킨다.

### v0.10.0 · 파일 기반 프로젝트

웹 상태만이 아니라 로컬 프로젝트 폴더를 진실의 출처로 만든다.

- `storyx init`: 프로젝트 폴더 + `memory-bank/` + `storyx.config.json` 생성.
- `storyx serve`: 생성된 폴더를 기준으로 웹 에디터 실행.
- `storyx memory sync`: 승인 큐의 approved 항목만 실제 `canon`/`characters`/`world`/`voice` 파일에 반영.
- CLI와 웹 에디터가 같은 프로젝트 상태와 메모리 뱅크를 읽는다.

완료 기준 — 새 폴더에서 `storyx init` 후 `storyx serve`로 에디터가 열리고, 승인 항목만 파일에 안전하게 쓰인다.

### v1.0.0-alpha · 첫 완성 글쓰기 루프

아이디어 → 작가 인터뷰 → 초안 생성 → 직접 편집 → 흐름 검증 → 바이블 승인 → 출간 스냅샷 잠금. 이 루프가 소설과 에세이에서 끊기지 않으면 첫 알파 완성 기준에 도달한다.

완료 기준 — 장편 기준 10화 이상 생성·수정해도 캐릭터 욕망·세계 규칙·사건 타임라인이 유지되고, 사용자가 직접 고친 문장을 AI가 무단 삭제하지 않는다.

## 4. 그 이후 — 베타에서 정식까지

- **매체별 제작**: 만화 컷 보드·말풍선·다빈치 이미지 프롬프트, 오디오북 낭독 보드, 매체 전환 브릿지(`novel-to-comic` 등).
- **비공개 베타**: 실제 사용자 프로젝트 10개 이상을 끝까지 제작하고 재미·일관성·검토 신뢰도를 측정.
- **공개 베타**: 크레딧·검토 규모별 사용량, 무료 생성 / 유료 깊은 검토·매체 전환, 퍼블리싱 패키지.
- **정식 출시**: 클라우드 저장, 팀 협업 권한, 장기 연재 메모리 버전 관리, 플랫폼별 출판 패키지.

## 5. 우선순위

- **P0** — 승인 루프 닫기(v0.9.0), `storyx init`/`serve`/`memory sync`, 직접 편집 diff, 장편 10화 회귀 하네스.
- **P1** — 매체 전환 브릿지, 다빈치 이미지 프롬프트 팩, 에세이 인터뷰·문체 일관성, 오디오북 낭독 보드, 프로젝트 버전 히스토리.
- **P2** — 결제·크레딧·유료 검토, 클라우드 동기화, 플랫폼별 출판 패키지, 팀 협업.

## 6. 베타 성공 기준

- 사용자가 CLI 한 줄로 프로젝트를 만들고 웹 에디터를 연다.
- 초안 생성보다 검토와 수정 경험이 더 믿음직하다.
- 장편 회차가 늘어나도 에이전트가 "무엇을 기억해야 하는지" 설명할 수 있다.
- 에세이에서는 AI가 없는 기억을 지어내지 않고 질문을 먼저 한다.
- 모든 새 캐논과 성장 메모리는 사용자 승인 뒤에만 저장된다.
