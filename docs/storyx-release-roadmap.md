# Story X Release Roadmap

> 작성 기준: 2026-05-14. 베타까지는 Claude Code/Codex CLI 같은 로컬 실행 방식을 우선한다. 최종 제품은 웹 서비스로 확장하되, 핵심 품질은 CLI 베타에서 먼저 증명한다.

## 1. 현재 검증 상태

현재 프로토타입은 다음 범위에서는 문제없이 동작한다.

- `npm test`: 87개 테스트 통과.
- `npm run build`: TypeScript와 Vite 프로덕션 빌드 통과.
- GitHub `main` 푸시와 Vercel 프로덕션 배포가 연결되어 있으며, 최신 알파는 `https://story-x-alpha.vercel.app`에서 확인 가능하다.
- 브라우저 스모크 테스트: 랜딩 -> 로그인 -> 프로젝트 -> 새 프로젝트 -> 에디터 -> 초안 생성 -> 직접 편집 -> 검토 흐름 확인.
- 새 프로젝트 플로우: 매체 선택 -> 성향 질문 -> 에디터 진입이 전체화면 좌우 전환 구조로 이어진다.
- 에이전트 레이어: 보이는 프론트 에이전트, 보이지 않는 미드 오케스트레이션, 백엔드/메모리 운영 에이전트가 단계별로 분리된다.
- 홈/랜딩 내비 분리: 같은 리퀴드 글라스 형태를 유지하되, 페이지별 내비 내용은 분리됨.
- 에디터 기본 구조: 왼쪽 이야기 형태, 중앙 창작물/편집기, 오른쪽 메모리 뱅크/품질 게이트/에이전트 의견 구조 확인. 1280px 폭에서도 3열 창작 구조를 유지한다.
- 원고 직접 편집: 생성된 원고를 편집할 수 있고, 수정 상태와 검토 버튼 흐름이 동작함.
- 회차 이동: 1화/2화 탭과 작품 목차 트리의 UI 기반이 들어감.
- 작품 바이블: story, character, world, canon/timeline, voice/visual/audio, approval 대기 트랙이 분리됨.
- 메모리 뱅크 작업대: editable records, role-specific context packets, private/raw-sources 제외 규칙을 UI와 도메인 모델로 확인함.
- 출간 화면: 출간 준비는 에디터 탭이 아니라 별도 액션이며, 출간 전 체크리스트, 출간 스냅샷, 변경 로그 검토, 오디오북 제안을 보여준다.
- AI CLI 하네스: mock/Claude/Codex provider 선택, 검토 규모 선택, provider raw output 정규화, `reviews/pending` 저장 계약을 테스트함.
- 메모리 승인 큐: 회차 canon 후보와 AI review `memoryCandidates`를 하나로 모으고, 편집/승인/수정 요청/보류/동기화 가능 여부를 표시함.

아직 "출시 품질"이라고 말하면 안 되는 범위는 다음과 같다.

- Claude/Codex provider 어댑터와 정규화기는 있으나, 실제 사용자 프로젝트에서 반복 실행한 비용/실패/재시도 검증은 아직 부족하다.
- `storyx init`, `storyx serve`, `storyx memory sync`, `storyx export`는 아직 베타 수준 명령으로 완성되지 않았다.
- 메모리 뱅크는 논리 모델, UI, 승인 큐가 있으나, 승인된 항목을 실제 프로젝트 폴더 파일에 안전하게 쓰는 `memory sync` 루프가 더 필요하다.
- 에이전트 검토는 mock과 provider output 계약을 검증한 단계이며, 실제 장편 프로젝트에서 에이전트별 품질 차이를 측정해야 한다.
- 장편 연재 수준의 장기 회귀 테스트, 한국어 문체 유지 평가, 매체 전환 산출물 검증은 아직 부족하다.
- 만화/오디오북은 workflow board와 메모리 구조가 있으나, 실제 컷 편집기, 이미지 프롬프트 팩, 오디오 라인 편집기는 아직 초기 설계 단계다.
- 결제, 계정, 클라우드 저장, 팀 협업, 퍼블리싱 패키지는 최종 서비스 단계 과제다.

## 1.1 현재 알파 판정

현재 상태는 "로컬 알파"로 보는 것이 맞다.

- UI 알파: 랜딩, 홈, 프로젝트, 에디터, 작품 바이블, 검토 레일, 승인 큐가 이어진다.
- 엔진 알파: story engine, memory bank, AI CLI harness, visual workflow, evaluator workflow가 테스트로 보호된다.
- 베타 전 미완성: 실제 파일 기반 프로젝트 초기화, 승인 후 파일 동기화, 실제 Claude/Codex 반복 실행, 장편 회귀 테스트, export package.

따라서 다음 목표는 "기능을 더 많이 보이게 하기"보다 "지금 보이는 흐름이 실제 로컬 프로젝트 파일을 읽고 쓰게 만들기"다. 다음 마일스톤은 UI 구조가 아니라 CLI 프로젝트 폴더, memory sync, review-ledger, 실패 로그를 실제 파일로 연결하는 것이다.

## 2. 제품 원칙

Story X의 출시 기준은 "많이 생성한다"가 아니라 "오래 가도 무너지지 않는 이야기를 만든다"이다.

- 이야기가 먼저이고, 매체는 그 다음이다.
- 원고 전체를 매번 컨텍스트로 밀어 넣지 않고, 메모리 뱅크에서 역할별 컨텍스트 패킷을 만든다.
- 캐논 충돌은 다수결로 통과시키지 않는다. reveal, revision, blocked draft 중 하나로 분류한다.
- 사용자가 직접 고친 문장은 AI가 덮어쓰지 않는다. 검토, 대안, 영향 범위를 먼저 제시한다.
- 새 기억과 새 캐논은 사용자 승인 뒤에만 반영한다.
- 베타는 로컬 CLI에서 완성도를 증명하고, 이후 웹 서비스로 확장한다.

## 3. 베타 실행 방식

베타는 `storyx` CLI와 웹 에디터를 함께 실행하는 구조로 간다.

### CLI 목표

Claude Code처럼 프로젝트 폴더를 중심으로 실행한다.

- `storyx doctor`: 환경, Node 버전, 테스트, Claude/Codex 사용 가능 여부 점검.
- `storyx init`: 새 Story X 프로젝트와 `memory-bank/`, `.claude/`, `storyx.config.json` 생성.
- `storyx serve`: Vite 웹 에디터 실행.
- `storyx project new`: 매체, 포맷, 장르, 독자 약속을 받아 프로젝트 생성.
- `storyx draft`: 쇼러너, 캐릭터, 세계관, 장르 에이전트 순서로 초안 생성.
- `storyx review --scale small|standard|deep`: 사용자가 누른 검토 규모에 맞춰 에이전트 검토 실행.
- `storyx memory sync`: 승인된 캐논, 문체, 캐릭터, 세계관, 시각/오디오 기억을 메모리 뱅크에 반영.
- `storyx export`: 원고, 목차, 캐논 장부, 프롬프트 팩, 오디오/영상 준비물을 묶어 출력.

### Claude Code/Codex 호환 방식

- Claude Code: `.claude/agents/*.md`와 `.claude/skills/*`를 네이티브 자산으로 사용한다.
- Codex: `AGENTS.md`, `docs/codex-agent-manifest.md`, `docs/storyx-memory-bank.md`를 읽어 같은 역할 체계를 적용한다.
- 공통: CLI는 어떤 실행 환경이든 같은 `memory-bank/`와 `storyx.config.json`을 기준으로 동작한다.
- 베타에서는 클라우드 DB보다 파일 기반 기록을 우선한다. 사용자가 결과를 직접 보고, 승인하고, 되돌릴 수 있어야 한다.

## 4. 출시 단계

### Phase 0. 프로토타입 안정화

상태: 완료에 가까움. 현재 알파 기준으로 UI 구조와 핵심 도메인 계약은 통과했다.

목표:

- 랜딩, 홈, 에디터 3단계가 명확히 분리된다.
- 새 프로젝트는 전체화면 플로우로 매체 선택, 질문 세팅, 에디터 진입을 순서대로 안내한다.
- 에디터 중앙이 창작물에 우선 배정되고, 좌우 레일은 보조 정보로 고정된다.
- 생성 후 직접 편집과 검토 버튼 흐름이 동작한다.
- 회차 탭과 작품 목차가 보인다.
- 출간 준비는 별도 액션으로 분리되고, 출간 뒤 수정 영향 범위를 검토한다.

완료 기준:

- `npm test`, `npm run build` 통과.
- 브라우저에서 기본 제작 플로우가 끊기지 않는다.
- 영어 내부 용어가 사용자 화면의 핵심 영역에 남지 않는다.
- 검토 결과가 승인 큐로 이동하고, 승인 전에는 메모리 뱅크에 반영되지 않는다.
- 프론트/미드/백 에이전트가 사용자가 보는 도움, 보이지 않는 오케스트레이션, 파일/메모리 운영으로 역할 충돌 없이 분리된다.

남은 정리:

- 일부 내부 용어와 밀도 높은 카드의 카피를 더 서비스 언어로 다듬는다.
- 작은 화면에서 에디터/바이블/검토 레일의 사용성을 한 번 더 확인한다.

### Phase 1. CLI 베타 하네스

상태: 부분 완료. `tools/storyx.mjs` 기반 알파 하네스, mock/provider review, raw output normalization, pending review 저장 계약은 들어갔다.

목표:

- `storyx` CLI를 만들고, 로컬 프로젝트에서 Claude Code처럼 실행 가능하게 한다.
- 웹 에디터와 CLI가 같은 프로젝트 상태와 메모리 뱅크를 읽는다.

구현 항목:

- `tools/storyx.mjs`를 베타 CLI의 임시 엔트리로 유지하되, 명령 구조가 커지면 `src/cli/` 또는 `tools/storyx-cli/`로 분리.
- `storyx.config.json` 스키마 정의.
- `memory-bank/` 실제 파일 생성, 읽기, 승인 대기, 동기화 명령 구현.
- `storyx doctor`, `storyx init`, `storyx serve`, `storyx review`부터 구현.
- 알파 우선순위는 `doctor`, `review --provider mock|claude|codex`, `normalize-provider-output`을 유지하고, 다음은 `init`, `serve`, `memory sync`다.
- CLI 테스트 추가.

완료 기준:

- 새 폴더에서 `storyx init` 후 `storyx serve`로 현재 웹 에디터가 열린다.
- `storyx review --scale small`이 에이전트별 검토 보고서를 만든다.
- 승인되지 않은 새 캐논은 메모리 뱅크에 쓰이지 않는다.
- `storyx memory sync`가 승인 큐의 approved 항목만 실제 memory-bank 파일에 반영한다.

### Phase 2. 쓰기방/검토방 베타

상태: UI/모델 일부 완료. 실제 반복 작업 품질 검증은 미완료.

목표:

- 쇼러너, 캐릭터, 세계관, 장르, 문체, 연속성 에이전트가 실제 제작 단계에서 충돌 없이 협업한다.
- 사용자가 검토 규모를 고르면 비용과 깊이에 맞춰 검토가 실행된다.

구현 항목:

- 에이전트 실행 순서: Story Contract -> Character/World Checks -> Genre/Voice Pass -> Continuity Gate -> User Approval.
- 검토 규모: 빠른 검토, 표준 검토, 깊은 검토.
- 직접 편집 구간 diff 표시.
- 검토 결과를 `review-ledger`와 `reviews/pending`에 저장.
- 사용자 승인 후에만 `canon`, `voice`, `character`, `world` 기억 갱신.
- AI Output Autopsy가 만든 memoryCandidates를 승인 큐에 합류시킨다.

완료 기준:

- 장편 기준 10화 이상 생성/수정해도 캐릭터 욕망, 세계 규칙, 주요 사건 타임라인이 유지된다.
- 사용자가 직접 고친 문장을 AI가 무단 삭제하지 않는다.
- 검토 보고서가 "문제", "대안", "반영 여부", "메모리 업데이트 후보"를 분리한다.

### Phase 3. 매체별 제작 워크플로우

상태: 소설/에세이/만화/오디오북의 선택 구조와 workflow board는 존재한다. 실제 매체별 편집기는 초기 프로토타입이다.

목표:

- 소설, 에세이, 만화, 오디오북이 각자 다른 화면과 검토 기준을 가진다.
- 하나의 이야기에서 다른 매체로 자연스럽게 전환된다.

구현 항목:

- 소설/웹소설: 회차 설계, 후크, 다음 화 질문, 캐논 장부.
- 에세이: 인터뷰 질문, 실제 인물 익명화, 문체 샘플, 민감 정보 보류.
- 만화: 컷 보드, 말풍선 밀도, 다빈치 이미지 프롬프트, 캐릭터 시각 바이블.
- 만화 추가 기준: 스토리보드, 말풍선 연출, Midjourney 원화/키프레임 선택, 다빈치 컷별 프롬프트, 프레임 조립을 분리한다.
- 오디오북/영상: 낭독 톤, 쉼표, 음악 큐, 자막 밀도, 장면별 오디오 보드.
- 매체 전환 명령: `storyx convert novel-to-comic`, `storyx convert essay-to-audio`, `storyx convert comic-to-video-board`.
- 변환 결과도 기존 story contract와 memory bank를 읽고, 새 visual/audio 후보는 승인 큐를 거친다.

완료 기준:

- 소설 한 회차를 웹툰 컷 보드와 동화책 페이지 보드로 변환할 수 있다.
- 에세이를 오디오북 낭독 스크립트로 변환할 때 개인정보와 문체가 보호된다.
- 만화 컷별 이미지 프롬프트가 캐릭터 외형과 시각 규칙을 유지한다.

### Phase 4. 비공개 베타

목표:

- 실제 사용자 프로젝트 10개 이상을 로컬 CLI/웹 에디터로 끝까지 제작한다.
- 재미, 일관성, 직접 편집성, 검토 신뢰도를 측정한다.

구현 항목:

- 프로젝트 라이브러리, 버전 히스토리, 복원.
- 산출물 패키지 다운로드.
- 실패 로그와 사용자 승인/거절 기록.
- 에이전트별 성장 메모리 리포트.
- 한국어 자연스러움 평가 루프.

완료 기준:

- 단편 3개, 장편 연재 샘플 2개, 에세이 3개, 만화/오디오 샘플 2개를 끝까지 만든다.
- 사용자가 "AI가 내 글을 빼앗았다"고 느끼는 순간을 줄인다.
- 검토 후 실제 수정률과 만족도가 기록된다.

### Phase 5. 공개 베타

목표:

- 로컬 우선 구조를 유지하되, 서비스형 사용 경험을 준비한다.
- 수익화 전환 지점을 검증한다.

구현 항목:

- 계정/프로젝트 동기화 선택 옵션.
- 크레딧/검토 규모별 사용량 표시.
- 무료 생성, 유료 깊은 검토, 유료 매체 전환 패키지 설계.
- 퍼블리싱/다운로드 패키지.
- 모델 제공자 어댑터: Claude, OpenAI, 이미지 모델, TTS 모델을 교체 가능하게 설계.

완료 기준:

- 무료 사용자는 로컬/기본 검토로 가치를 느낀다.
- 유료 사용자는 깊은 검토, 장편 메모리, 매체 전환에서 지불 이유를 느낀다.
- 실패 시 비용이 새지 않고, 중간 결과가 보존된다.

### Phase 6. 정식 출시

목표:

- Story X를 "이야기 운영체제"로 출시한다.
- 사용자는 소설, 에세이, 만화, 오디오북, 영상 보드 사이를 오가며 하나의 작품을 관리한다.

구현 항목:

- 클라우드 프로젝트 저장과 로컬 내보내기 병행.
- 팀/작가/편집자 협업 권한.
- 장기 연재용 메모리 뱅크 버전 관리.
- 플랫폼별 출판 패키지.
- 보안, 백업, 결제, 고객 지원, 법무/개인정보 정책.
- 품질 게이트와 사용량 기반 과금.

완료 기준:

- 장편/시리즈 프로젝트가 50화 이상에서도 캐릭터, 세계관, 문체가 유지된다.
- 사용자가 직접 편집한 내용과 AI 제안이 명확히 구분된다.
- 한 작품을 다른 매체로 전환해도 핵심 story contract가 유지된다.

## 5. 우선순위

P0:

- `storyx init` / `storyx serve` / `storyx memory sync`.
- 승인 큐 -> 실제 memory-bank 파일 반영.
- 실제 Claude/Codex small review 반복 검증과 실패 복구.
- 직접 편집 diff와 review-ledger 저장.
- 장편 10화 회귀 테스트 하네스.

P1:

- 매체 전환 브릿지.
- 다빈치 이미지 프롬프트 팩.
- 에세이 인터뷰/문체 일관성.
- 오디오북 낭독 보드.
- 프로젝트 버전 히스토리.
- Platform Packaging Lab: 웹소설 첫 문장, 웹툰 첫 3컷, 인스타툰 저장 컷, 오디오북 첫 30초.

P2:

- 결제, 크레딧, 유료 검토.
- 클라우드 동기화.
- 플랫폼별 출판 패키지.
- 팀 협업.

## 6. 다음 구현 순서

1. `storyx init`: 새 프로젝트 폴더, `storyx.config.json`, `memory-bank/`, `.claude/` 호환 자산을 생성한다.
2. `storyx serve`: 생성된 프로젝트 폴더를 기준으로 웹 에디터를 연다.
3. `storyx memory sync`: 승인 큐의 approved 항목만 실제 `canon`, `characters`, `world`, `voice`, `visual`, `audio` 파일에 반영한다.
4. `review-ledger`와 `failure-log`를 실제 파일로 저장하고, 에디터 오른쪽 레일에서 불러온다.
5. 실제 `claude --print` / `codex exec` small review를 같은 샘플로 반복 실행하고 실패/비용/정규화 로그를 남긴다.
6. 웹 에디터에서 CLI가 만든 `reviews/pending` 파일을 불러와 승인 큐에 표시한다.
7. 직접 편집 diff를 줄 단위가 아니라 변경 블록 단위로 표시하고, 검토 요청 시 diff만 에이전트 패킷에 포함한다.
8. 소설 -> 만화/동화책/오디오북 변환 보드를 만들고, visual/audio 후보도 승인 큐를 통과시킨다.
9. 장편 10화, 에세이 3편, 인스타툰 2편, 오디오북 1편의 로컬 베타 샘플을 제작한다.
10. 가격/크레딧/출판 패키지를 붙여 공개 베타로 간다.

## 7. 베타 성공 기준

베타는 다음을 만족해야 한다.

- 사용자가 CLI 한 줄로 프로젝트를 만들고 웹 에디터를 열 수 있다.
- 초안 생성보다 검토와 수정 경험이 더 믿음직해야 한다.
- 장편 회차가 늘어나도 에이전트가 "무엇을 기억해야 하는지" 설명할 수 있다.
- 에세이에서는 AI가 없는 기억을 지어내지 않고 질문을 먼저 한다.
- 만화와 오디오북은 단순 변환이 아니라 컷, 소리, 리듬의 구조를 제안한다.
- 모든 새 캐논과 성장 메모리는 사용자 승인 뒤에만 저장된다.
