# Story X 계획 — 제품 완성에서 서비스 런칭까지

> 작성 2026-05-16 · 기준 버전 `Alpha v0.8.0`.
> **이 문서가 Story X 계획의 단일 출처다.** 과거 문서(`story-x-prd.md`, `agent-workflow-plan.md`, `storyx-evaluator-development-update.md`)는 배경 자료다. 버전별 변경 기록은 `storyx-version-log.md`.

## 0. 한 줄 정의

Story X는 창작자가 만든 하나의 이야기가 형태(소설·에세이·만화·오디오북·영상)를 바꿔도 영혼을 잃지 않게 돕는, 이야기 우선 창작 운영체제다.

## 1. 두 단계

개발은 두 단계로 나눈다.

- **1단계 — 만족스러운 작품이 나오는 제품.** 만든 사람이 claude 구독으로 자유롭게 써서, 끝까지 만족스러운 작품이 생성·검토·발전되는 상태. 외부 사용자나 서비스 인프라는 아직 생각하지 않는다. **제품 완성도는 이 단계에서 거의 끝낸다.**
- **2단계 — 공개와 상품화.** 1단계에서 완성된 제품을 외부 창작자도 쓸 수 있게 만들고, 서비스로 출시·과금한다.

핵심 원칙 — **완성도는 1단계에서 거의 마무리한다. 2단계는 새 기능을 더하는 단계가 아니라, 완성된 제품을 남에게 팔 수 있게 포장하는 단계다.** 그래서 서버측 LLM·계정·결제 같은 서비스 인프라는 전부 2단계로 미룬다. 1단계 내내 LLM은 로컬 claude 구독으로 충분하다 — 쓰는 사람이 한 명이기 때문이다.

## 2. 지금 상태 — v0.8.0

**되는 것**
- 랜딩 → 로그인 → 프로젝트 → 새 프로젝트 → 에디터 흐름, 편집/바이블 2트랙 에디터.
- 초안 생성·작가진 검토가 실제 Claude 호출로 동작(로컬 dev 브리지 경유).
- 2화 이상 생성 시 확정 캐논·인물·세계를 LLM에 넘겨 연속성 유지.
- 메모리 승인 큐, 캐논/캐릭터/세계/문체 바이블, ⌘K 명령 팔레트, `storyx` CLI 하네스.

**아직 아닌 것 (1단계에서 끝낼 것)**
- 검토에서 승인한 기억이 실제 작품 캐논으로 반영되는 루프가 닫히지 않았다.
- 장편 회귀 검증, 한국어 문체 유지 평가가 없다.
- 작품이 영속적으로 저장·복원되지 않는다.
- 만화 컷 편집기·오디오 라인 편집기는 초기 설계 단계다.

## 3. 1단계 — 제품 완성 (지금 집중)

### 1단계 완료 기준

> "내가 Story X로 자유롭게 작품을 쓰는데, 생성·검토·수정·연속성이 끝까지 만족스럽다."

구체적으로 — 장편 10화 이상을 만들어도 캐릭터·세계·캐논이 무너지지 않고, 검토가 믿을 만하고, 내가 직접 고친 문장을 AI가 덮지 않고, 작품이 저장되어 다음에 이어 쓸 수 있다.

### A · 핵심 창작 루프 닫기 — `v0.9.0` (다음 작업)

검토가 만든 기억 후보가 승인을 거쳐 실제 작품 상태로 들어가게 한다. 이게 닫혀야 생성·검토·연속성이 한 루프가 된다.

- 검토 → `memoryCandidates` → 승인 대기함 → 사용자 승인 → `project`의 canon/character/world 갱신.
- 갱신된 캐논이 다음 회차 생성 컨텍스트에 자동 반영.
- 직접 편집 diff를 변경 블록 단위로 표시하고, 검토 패킷에는 diff만 포함.
- `review-ledger`와 실패 로그 기록.

완료 — 회차를 생성·검토하고 후보를 승인하면 바이블에 반영되고, 다음 생성이 그 캐논을 지킨다.

### B · 작품 품질과 장편 연속성 — `v0.10.0`

작품이 "만족스럽다"의 핵심. 길어져도 무너지지 않게 한다.

- 장편 10화 이상 회귀 하네스 — 회차가 쌓여도 캐릭터 욕망·세계 규칙·타임라인 유지 검증.
- 메모리 뱅크 요약 품질 개선 — 컨텍스트가 커질 때 무엇을 남기고 무엇을 줄일지.
- 한국어 문체 유지 평가 루프 — 번역투·AI투를 잡는다.
- 검토 신뢰도 — 작가진 5종 판정이 실제로 쓸 만한 수정 지시인지 점검.

완료 — 장편 표본 1편을 끝까지 써서 연속성·문체·검토가 만족스럽다.

### C · 작품 영속성 — `v0.11.0`

개인 사용에 필요한 만큼만. 서버 계정이 아니라 로컬·파일 기반으로.

- 작품이 저장되고 다음에 그대로 열린다(캐논·회차·승인 상태 포함).
- `storyx init`/`serve`/`memory sync`로 로컬 프로젝트 폴더와 웹이 같은 상태 공유.
- 작품 버전 히스토리와 복원.

완료 — 작품을 닫았다 다시 열어도 모든 상태가 그대로다.

### D · 매체별 제작 — `v0.12.0`

소설·에세이를 먼저 만족스럽게 끝내고, 만화·오디오로 넓힌다.

- 소설/에세이 제작·검토 완성도 마무리.
- 만화 컷 보드·말풍선·다빈치 이미지 프롬프트, 오디오북 낭독 보드.
- 매체 전환 브릿지(`novel-to-comic` 등) — 같은 Story Contract 유지.

완료 — 소설·에세이는 끝까지, 만화·오디오는 한 편씩 만들어 본다. **여기까지가 1단계 = 개인용 완성(v1.0).**

## 4. 2단계 — 공개와 상품화

1단계에서 완성된 제품을 외부 창작자가 쓰게 만들고 서비스로 출시한다. 새 창작 기능이 아니라 서비스 인프라와 사업화가 중심이다. 여섯 단계로 나눈다.

### E · 서버측 LLM 운영

목표 — 외부 사용자가 claude CLI 없이 브라우저만으로 생성·검토를 쓴다.

핵심 작업
- `/api/draft`·`/api/review`를 dev 브리지가 아니라 Vercel Function(서버)으로 구현.
- Anthropic API 연동, 모델·토큰 한도·타임아웃 설정.
- 실패·타임아웃·재시도 처리, 부분 결과 보존.
- 동시 요청 큐잉과 사용자별 rate limit.
- (선택) BYO — 사용자가 자기 API 키나 구독 토큰을 연결하는 경로.

완료 기준 — 로그인한 외부 사용자가 브라우저만으로 초안 생성과 검토를 끝까지 받는다.

### F · 계정과 클라우드 저장

목표 — 작품이 기기가 아니라 계정에 속한다.

핵심 작업
- 계정·로그인(이메일 또는 소셜), 세션 관리.
- 프로젝트·버전 스냅샷·승인 상태를 클라우드 DB에 저장. 현 localStorage 모델을 이관.
- 기기 간 동기화, 작품 목록과 복원.
- 로컬 우선 원칙 유지 — 내보내기로 사용자가 자기 데이터를 가져갈 수 있다.

완료 기준 — 다른 기기로 로그인해도 작품·캐논·버전 기록이 그대로 열린다.

### G · 과금 구조

목표 — 무료 가치와 유료 가치의 경계를 만들고, 비용이 새지 않게 한다.

핵심 작업
- 크레딧 또는 구독 모델 설계 — 무료 생성 한도, 유료 깊은 검토·매체 전환.
- 결제 연동(Stripe 등), 사용량·잔여 크레딧 표시.
- 비용 가드레일 — 실패 시 과금 안 함, 1회 호출 상한, 남용 차단.
- LLM 원가 대비 가격 검증으로 적자 런칭을 막는다.

완료 기준 — 무료 사용자도 가치를 느끼고, 유료 전환 지점이 분명하며, 호출당 비용이 통제된다.

### H · 비공개 베타

목표 — 실제 창작자에게서 제품이 버티는지 확인한다.

핵심 작업
- 초대 기반 비공개 베타, 외부 창작자 10명 이상.
- 단편·장편·에세이 표본을 끝까지 제작.
- 재미·일관성·직접 편집성·검토 신뢰도 측정, 실패 로그·이탈 지점 수집.
- 피드백을 다음 개발로 되돌리는 주기.

완료 기준 — "AI가 내 글을 빼앗았다"는 순간이 거의 없고, 검토 후 수정률·만족도가 기록된다.

### I · 공개 베타

목표 — 누구나 가입해 쓰는 규모로 연다.

핵심 작업
- 공개 가입, 대규모 온보딩 — 첫 작품 완성까지 막힘 없이 안내.
- 고객 지원·문의 채널, 서비스 상태 페이지.
- 부하·비용 모니터링과 장애 대응.
- 약관·개인정보 처리방침·저작권 귀속 1차 정리(특히 에세이 민감정보).

완료 기준 — 초대 없이 가입한 사용자가 도움 없이 첫 작품을 만든다.

### J · 정식 런칭

목표 — Story X를 이야기 운영체제로 정식 출시한다.

핵심 작업
- 플랫폼별 출판·내보내기 패키지(웹소설·웹툰·오디오 등).
- 팀·작가·편집자 협업 권한.
- 장기 연재용 메모리 뱅크 버전 관리.
- 보안·백업·법무·결제 운영 안정화, 런칭 마케팅.

완료 기준 — 1조의 런칭 게이트 7개 항목이 모두 참이고, 외부 창작자가 작품을 만들어 유료로 머문다.

## 5. 우선순위

- **현재** — 1단계 A·B·C와 D의 소설·에세이 경로는 완료. 남은 1단계 작업은 D의 만화·오디오 갈래와, 만든 사람의 실사용 품질 평가.
- **다음(P0)** — 2단계 E(서버측 LLM). 외부 사용자가 없으면 서비스가 아니다. 서비스 성립의 전제.
- **P1** — F(계정·클라우드 저장), G(과금 구조).
- **P2** — H(비공개 베타) → I(공개 베타) → J(정식 런칭).

## 6. 제품 원칙 (불변)

- 이야기가 먼저이고, 매체는 그 다음이다.
- 원고 전체를 매번 컨텍스트로 밀어 넣지 않고, 메모리 뱅크에서 역할별 컨텍스트 패킷을 만든다.
- 캐논 충돌은 다수결로 통과시키지 않는다. reveal / revision / blocked 중 하나로 분류한다.
- 사용자가 직접 고친 문장은 AI가 덮어쓰지 않는다. 검토·대안·영향 범위를 먼저 제시한다.
- 새 기억과 새 캐논은 사용자 승인 뒤에만 반영한다.

## 7. 리스크

- **장편 연속성** — 회차가 쌓일수록 컨텍스트가 커진다. 메모리 뱅크 요약 품질이 1단계 성패를 가른다.
- **LLM 비용 구조** — 2단계 서버측 생성은 토큰 비용이 발생한다. 무료 한도와 유료 경계를 E~H에서 같이 설계해야 적자 런칭을 피한다.
- **개인정보** — 에세이 매체는 실제 인물·민감 기억을 다룬다. 익명화와 보관 정책이 2단계 공개 전 필수다.
- **창작자 신뢰** — AI가 사용자의 글을 덮어쓰는 경험이 한 번이라도 나오면 이탈한다. 직접 편집 보호가 최우선 품질 기준이다.
